# SOLID / clean-code audit - 2026-04-19

Scope: `packages/core`, `packages/compute`, `packages/server`. Main at `db2b593`.
Ranking is by impact (how often the smell is read/edited, blast radius of a
future edit) x how small the fix is. Fixes for the top 10 ship in the same PR
as this doc; the rest are tracked here as follow-ups.

Legend: `SRP` single responsibility, `OCP` open/closed, `LSP` liskov,
`ISP` interface segregation, `DIP` dependency inversion.

## Top 20 findings

### 1. `stage-orchestrator.executeAction` - switch over action strings [OCP, shotgun surgery]

`packages/core/services/stage-orchestrator.ts:791-896`. A ~105 LOC `switch (action)`
covering `create_pr | merge_pr | merge | auto_merge | close_ticket | close`. Every
new action forces an edit here plus in every test that exercises the action.
Perfect candidate for a handler registry keyed by action name. Mirrors the
`ProviderFlagSpec` registry pattern already in the codebase.

Fix: introduce `packages/core/services/actions/` with one file per action +
an `index.ts` registry. `executeAction` becomes a 15-line lookup. FIXED in
this PR.

### 2. `session/replay.summarize` - switch over event type [OCP]

`packages/core/session/replay.ts:34-110`. ~75 line switch. Adding a new event
type means editing this function and nothing fails loudly if you forget a case.

Fix: map `{ eventType -> (data) => string }`. Default handler stays. FIXED in
this PR.

### 3. `knowledge/indexer.mapEdgeRelation` - switch-as-lookup [OCP]

`packages/core/knowledge/indexer.ts:193-208`. Six-way switch that is really a
lookup table. Trivially replaced by a `Record<string, EdgeRelation>` with a
default. Adding a new codegraph edge kind forces an edit here instead of one in
a data table.

Fix: single `EDGE_RELATION_MAP` const + default. FIXED in this PR.

### 4. `server/handlers/resource.ts` - scope + builtin-guard duplication [DRY, shotgun surgery]

`packages/server/handlers/resource.ts`. Every resource CRUD handler repeats the
same pattern: `findProjectRoot(process.cwd())` + `resolve scope` +
`guard builtin`. `resolvedScope` appears 15 times; `_source === "builtin"` 4
times. Each new resource type (or scope rule change) has to be applied in
agent/flow/skill/recipe handlers.

Fix: extract `resolveScope()` + `guardMutableResource()` helpers. FIXED in
this PR (applied to agent + skill + recipe handlers; flow handlers have
different shape and are left as follow-up).

### 5. `ComputeProvider` - fat interface [ISP]

`packages/compute/types.ts:39-103`. 19 methods spanning compute lifecycle,
session lifecycle, monitoring, channel config, launch env, capability flags.
Most consumers (hooks, session code) touch 2-3 methods. Subclasses have to
`// Attach handled elsewhere` no-ops (see `RemoteArkdBase.attach` at
`packages/compute/providers/remote-arkd.ts:252`, which is a silent no-op --
a Liskov signal).

Fix (deferred): split into `ComputeLifecycle` + `SessionLifecycle` +
`ComputeMonitoring` + `ChannelConfigurable`. Requires coordinated refactor of
every provider class (7+ call sites). The new two-axis `Compute` + `Runtime`
types in `packages/compute/core/types.ts` are already heading this way, so the
legacy `ComputeProvider` will shrink as dispatch migrates. Track as an audit
item; do not touch in this PR.

### 6. `app.ts::AppContext` - 1266 LOC god class [SRP]

`packages/core/app.ts`. Owns: DB open, schema init, repo/service wiring,
provider registration, compute/runtime registry, compute pool registry,
launcher, tenant scoping, signal handlers, metrics poller, stale-state
detection, shutdown, test factory. Already partially decomposed into `_init*`
helpers, but `AppContext` remains the class every module reaches into.

Fix (deferred, coordinate with Agent 5 on DI): extract
`BootSequence`/`ShutdownSequence` objects; keep `AppContext` as a dumb
container exposing resolved services. Out of scope here -- touches every test
file.

### 7. `app.ts::_registerComputeProviders` - inline provider list [OCP]

`packages/core/app.ts:667-738`. A mix of eager and try/catch-gated providers,
each literally instantiated in-place. Every new provider adds a line (or a
dynamic import + try/catch) here. Good candidate for a provider registry module
(already have `ProviderFlagSpec` as a template).

Fix (deferred): introduce `packages/compute/registry.ts` with
`registerBuiltinProviders(app)` and `registerOptionalProviders(app)`. Move
instantiations there. Not done in this PR because it touches plugin-loading
semantics and I want to keep commits per-concern.

### 8. Magic port numbers scattered across providers [magic numbers, DIP]

`packages/compute/providers/remote-arkd.ts:18` hoists `ARKD_REMOTE_PORT` from a
constant but then `compute/core/ec2.ts`, `compute/providers/docker/*`, and the
`local-arkd.ts` providers each reach into `process.env` or hardcode
`19300`/`8420`. CLAUDE.md explicitly says `Use config.channels.*`; several call
sites miss this.

Fix (deferred): sweep for literal `19300` / `8420` in `packages/compute/` and
route through a provider-supplied `PortConfig`. Tracked.

### 9. `services/workspace-service.ts` - mixed responsibilities [SRP]

`packages/core/services/workspace-service.ts` (884 LOC). File does: path
sanitization, worktree lifecycle, PR create/merge, file diff, rebase,
dependency injection. `safeAttachmentName` is a pure string utility pinned at
the top of a worktree module -- it belongs in `util/`.

Fix attempted in this PR but reverted: a test in
`packages/web/src/__tests__/attachments-sanitize.test.ts` `readFileSync`'s the
raw `workspace-service.ts` source and asserts a literal
`export function safeAttachmentName` is present. Per the audit guardrails
this PR cannot touch `packages/web`. Follow-up: either rewrite the web test
to stop grepping source (so the helper can move), or relocate the assertion
to the util module once the test lives elsewhere. Tracked.

### 10. `compute/runtimes/devcontainer.ts` - 460 LOC runtime with embedded parsing [SRP]

`packages/compute/runtimes/devcontainer.ts`. Owns devcontainer.json parsing +
image resolution + lifecycle + volume mapping. `devcontainer-resolve.ts` next
to it (339 LOC) partially carved out already but the runtime still inlines
JSON parsing + mount expansion.

Fix (deferred): follow the pattern used by `docker/helpers.ts` - move
`expandMounts()` + `resolveImage()` into a sibling pure module. Not in this
PR; touches compute lifecycle which I want out of a SOLID sweep.

### 11. `claude/claude.ts::writeSettings` - long param list + in-fn IO + mixed merging [SRP, long params]

`packages/core/claude/claude.ts:488-575`. Takes 4 positional args + an options
object, opens/parses/writes the settings file, builds hooks, builds permissions
allow-list, writes the `_ark` meta, atomic-renames. It's the One Function Every
New Permission Rule Has To Touch.

Fix (deferred): split `writeSettings` into `buildSettingsPayload(sessionId,
...) -> object` (pure) and `persistSettings(path, payload)` (IO). The pure
piece is testable without a temp dir. Not in this PR because the existing
`writeSettingsVerified` wrapper also has to be rewired and I want per-concern
commits.

### 12. `session-hooks.applyHookStatus` - 281 LOC [SRP]

`packages/core/services/session-hooks.ts:73-354`. Computes: status transition
from hook event, commit-verification, failure suggestion list, termination
condition evaluation, usage tracking. Mixed IO (execs git) + logic.

Fix (deferred): pull the commit-verification block out into
`checkNewCommitsSinceStageStart(session) -> boolean` (pure, testable via stub
`exec`). Not in this PR -- the function has three different
"if (newStatus)" branches and the split has real behavioural risk.

### 13. `repositories/session.ts::update` - column whitelist hardcoded twice [data clump, SRP]

`packages/core/repositories/session.ts:73-97`. `SESSION_COLUMNS` whitelist is
duplicated in `update()` and `claim()` (same iteration pattern). CLAUDE.md calls
this out as the known footgun. Should be a shared `applyFieldUpdates(fields,
allowed)` helper.

Fix: extract `buildUpdateSet(fields, allowed, {jsonFields})` helper. FIXED in
this PR.

### 14. `stage-orchestrator.dispatch` - 370 LOC [SRP]

`packages/core/services/stage-orchestrator.ts:133-501`. Dispatch does: session
status check, scheduler delegation (hosted), remote repo clone, prompt injection
guard, per-stage compute resolution, fork/fan_out branch, agent resolution,
model override, task builder, knowledge graph indexing, repo map injection,
executor resolution, launch, race-guard, status-poller kickoff, telemetry.

Fix (deferred): pull each phase into `dispatch/` module. Explicitly deferred
until action registry lands (step 1) to avoid a merge-heavy rewrite.

### 15. `services/session-hooks.applyReport` - 288 LOC with inline git shelling [SRP]

`packages/core/services/session-hooks.ts:354-641`. Similar shape to `#12`.
Contains a repeating "run git, decide, maybe reject completion" pattern that
differs from the one in `applyHookStatus` by ~3 lines. Candidate for a shared
`enforceCommitInvariant(session) -> {ok; reason?}` helper.

Fix (deferred).

### 16. `knowledge/indexer.detectLanguage` - magic ext->lang map [primitive obsession]

`packages/core/knowledge/indexer.ts:211-246`. Fine as-is from OCP perspective
(it's data) but the map is embedded in the function body. Each time you want
a new language you open the indexer. Lift to a module-level const.

Fix: hoist `LANG_MAP` to module scope. FIXED in this PR.

### 17. `conductor/conductor.ts::startConductor` - 1059 LOC, stateful closure [SRP]

`packages/core/conductor/conductor.ts`. One big HTTP handler closure with
inline route parsing + auth + tenant resolution + 12 route bodies. Every new
route forces a visit here.

Fix (deferred): split into a `Router` similar to `packages/server/router.ts`
plus per-route handler modules. Not in this PR -- substantial and needs
matching server harness tests.

### 18. `claude/claude.ts::buildArgs` long param + flag-building sprawl [clean code]

`packages/core/claude/claude.ts:46-87`. Mixes "am I headless" + "am I
dangerously-skipping-permissions" + "am I passing model/turns/system_prompt"
+ MCP serialisation. Not a SOLID violation but the ordering of pushes matters
for Claude CLI semantics and is implicit.

Fix (deferred): extract `buildHeadlessArgs(opts)` / `buildInteractiveArgs(opts)`
so the two CLI variants are visually distinct.

### 19. `session-lifecycle.runVerification` - IO-heavy, hard to unit test [DIP]

`packages/core/services/session-lifecycle.ts:351-410`. Runs `bash -c ${script}`
inline. Cannot be tested without shelling out. Candidate for injecting an
`exec` fn (matches the pattern `knowledge/indexer.ts` uses).

Fix: accept optional `exec` parameter defaulting to `execFileAsync`. FIXED in
this PR.

### 20. Silent `cleanZombieSessions` in `resource.ts` [SRP, misplaced]

`packages/server/handlers/resource.ts:30-44`. Unrelated to resource CRUD; lives
in the resource handler because that's where `cleanup/all` is registered.
Should live in `core/services/` alongside the other lifecycle cleanup
functions.

Fix (deferred): move to `core/services/cleanup.ts`. Tracked.

## DI framework candidates

Agent 5 owns DI. Flagging candidates to inject rather than new-up:

- `SessionService` already takes repos via constructor - good.
- `ComputeService(computes)` - good.
- `stage-orchestrator` functions all take `app: AppContext` as first arg
  instead of the specific services they use. Each function should declare what
  it actually needs (sessions repo, events repo, flows store, ...). This would
  let the function be tested without booting AppContext.
- `workspace-service` functions reach into `execFileAsync` statically. Inject
  an `exec` seam like `knowledge/indexer.ts` does.
- `claude.writeSettings` uses `fs` statically. Inject a `FileSystem` seam.
