Troubleshooting

Common issues, diagnostic tools, and solutions.

Diagnostic Tools

Start here when something goes wrong:

# Run the health check wizard
$ ark doctor

# Check session status and recent events
$ ark session read <session-id>

# View session events for debugging
$ ark session events <session-id>

# Attach to a running agent's tmux session
$ tmux attach -t ark-s-<session-id>

# List all ark tmux sessions
$ tmux list-sessions | grep ark-

Installation Issues

Bun not found

Ark requires Bun as its runtime. Install it:

$ curl -fsSL https://bun.sh/install | bash

If you installed via the self-contained tarball (install.sh), Bun is bundled -- make sure /usr/local/bin/ark is in your PATH.

tmux not found

Ark uses tmux to run agent sessions. Install it:

# macOS
$ brew install tmux

# Ubuntu/Debian
$ sudo apt install tmux

If you installed via the self-contained tarball, a static tmux binary is bundled in the Ark package.

"ark: command not found" after make install

The symlink may not be on your PATH. Check:

$ ls -la /usr/local/bin/ark
$ echo $PATH | tr ':' '\n' | grep local

If /usr/local/bin is not on your PATH, add it to your shell profile.

Session Issues

Session stuck in "running" state

The agent process may have exited without reporting back. Check:

  1. See if the tmux session still exists: tmux list-sessions | grep ark-s-<id>
  2. If the tmux session is gone, the agent crashed. Stop and restart:
$ ark session stop <session-id>
$ ark session dispatch <session-id>

Session stuck in "waiting" state

The session is waiting for a gate to be approved. Check what gate is pending:

$ ark session read <session-id>

Advance manually:

$ ark session advance <session-id>

Or force-advance (skips verification):

$ ark session advance <session-id> --force

Agent not receiving messages

Check that the channel relay chain is intact:

  1. ArkD is running: curl http://localhost:19300/health
  2. Conductor is running: curl http://localhost:19100/health (only if using the server daemon, web UI, or desktop app)
  3. Channel port is reachable. The port is derived from the session ID.

Verification scripts failing

If a stage has verify scripts, the agent cannot complete until they all pass. Check which script is failing:

$ ark session verify <session-id>

Fix the issue in the worktree, then re-run verification or force completion:

$ ark session complete <session-id> --force

Database Issues

Database locked errors

SQLite has a 5-second busy timeout. If you see "database is locked" errors, check for multiple Ark processes accessing the same database:

$ ps aux | grep ark

Only one server daemon/conductor instance should run at a time. Multiple CLI commands are fine (they use skipConductor: true).

Schema mismatch after upgrade

Ark does not have formal migrations (pre-pilot). If a column type changed or was renamed, delete the database and let it recreate:

$ rm ~/.ark/ark.db
$ ark web  # tables recreate from schema.ts on boot
Warning: This deletes all session history, events, and messages. Only do this during development -- not in production.

FTS5 search returning nothing

The full-text search index may not be built. Rebuild it:

$ ark index

Or from the web dashboard's History view, click the refresh button to rebuild the index.

Compute Issues

Docker provider: permission denied

Your user needs to be in the docker group:

$ sudo usermod -aG docker $USER
# Log out and back in for the group change to take effect

EC2 provider: instance not reachable

Check that the security group allows inbound traffic on port 19300 (ArkD). The EC2 provider sets up security groups automatically, but VPC or firewall rules may block traffic.

Worktree conflicts

If a worktree fails to create (e.g. because the branch already exists or the worktree path is occupied):

# List existing worktrees
$ git worktree list

# Clean up stale worktrees
$ git worktree prune

# Or use Ark's cleanup command
$ ark worktree cleanup

LLM Router Issues

Router not starting

Check that port 8430 is not already in use:

$ lsof -i :8430

If another process is using the port, either stop it or configure the router to use a different port:

$ ark router start --port 8431

All providers in circuit-breaker open state

If all LLM providers have tripped their circuit breakers, the router cannot route any requests. Check your API keys and provider availability. Circuit breakers auto-recover after 30 seconds.

$ ark router status

TensorZero fails to start

TensorZero tries three startup methods in order: sidecar detect, native binary, Docker. If all three fail:

  1. Install the native binary: make vendor-tensorzero (requires Rust/cargo)
  2. Or ensure Docker is running for the Docker fallback
  3. Or disable TensorZero in config: set tensorZero.enabled: false in ~/.ark/config.yaml

Hook Status Issues

Agent status not updating

Ark uses Claude Code hooks for agent status detection. If status is not updating:

  1. Check that .claude/settings.local.json exists in the session's working directory
  2. Verify the conductor is running and reachable at http://localhost:19100
  3. Check the hook endpoint: curl http://localhost:19100/hooks/status

Hooks config not cleaned up

When a session stops, Ark should remove .claude/settings.local.json. If it persists:

$ rm .claude/settings.local.json

This file is session-specific and safe to delete when no session is running in that directory.

Knowledge Graph Issues

Codegraph indexer not found

Install ops-codegraph:

$ npm install -g @optave/codegraph
# Or as a project dependency
$ bun add @optave/codegraph

Index taking too long

Large repos can take a while to index. The indexer uses tree-sitter (WASM) to parse 33 languages. For very large monorepos, consider indexing incrementally or indexing specific subdirectories.

Getting Help

If you're still stuck:

  • Run ark doctor and include the output in your bug report
  • Check session events: ark session events <id>
  • Search existing issues: GitHub Issues
  • File a new issue with reproduction steps
