Ark vs Goose from Block

How Ark differs from Block's Goose AI agent framework.

TL;DR

Ark is an orchestration platform. Goose is a standalone agent runner. Ark manages many sessions across many compute targets in parallel, with DAG-based SDLC flows, a unified knowledge graph, universal cost tracking, a multi-tenant control plane, and a pluggable runtime layer that can drive Claude Code, Codex, Gemini CLI, or any subprocess. Goose runs a single coding agent at a time on your local machine using recipe-based configuration. Both are useful; they solve different problems at different scales.

Goose is open source from Block (the company behind Square and Cash App). It is a good tool for individual developers. Ark is aimed at teams and platforms that need to orchestrate many agents across many compute targets with shared cost, knowledge, and policy state.

Architecture differences

A few specific architectural decisions enable the features that Goose does not have.

Runtime / role separation

Ark decouples what an agent does (its role -- planner, implementer, reviewer, etc.) from how it runs (its runtime -- Claude Code, Codex CLI, Gemini CLI, or any subprocess). Agent YAML specifies the role; runtime YAML specifies the backend. The same implementer role can run on Claude on one session and on Codex on the next with a single --runtime flag:

$ ark session start --summary "Fix bug" --agent implementer --runtime claude
$ ark session start --summary "Fix bug" --agent implementer --runtime codex
$ ark session start --summary "Fix bug" --agent implementer --runtime gemini

Goose binds the agent to its LLM provider directly. Swapping backends means editing the recipe or reinstalling.

Compute provider abstraction

Ark's ComputeService provisions a session's execution environment through a typed interface. The same session definition can run locally, in Docker, in a devcontainer, in a Firecracker VM, on EC2, or in a K8s pod -- with no code change. You set --compute k8s and the scheduler handles the rest. Goose runs on the local machine, full stop.

Universal agent daemon (ArkD)

Every compute target runs ArkD, a stateless HTTP server on port 19300. That means Ark does not need SSH or provider-specific transport to talk to remote compute. The conductor talks HTTP to ArkD; ArkD launches, observes, and reports back. Goose has no equivalent -- there is no remote compute to manage.

DI-based AppContext

Ark is built around an Awilix dependency injection container. AppContext is the root: services, repositories, stores, and the knowledge graph all resolve from it. This enables two things:

  • Tenant scoping. app.forTenant(tenantId) creates a scoped view of the same codebase for a different tenant. No globals, no leaks.
  • Testability. Every orchestration function takes app: AppContext as its first argument. Test isolation is a per-function concern, not a global state problem.

Goose is a single-user app and does not need this. But when you start thinking about multi-tenant hosting, this architecture is what lets you do it safely.

Migrating from Goose

A Goose recipe maps reasonably cleanly onto an Ark recipe plus agent plus flow. The concepts line up:

Goose conceptArk equivalent
Recipe (YAML preset)Recipe (YAML preset in recipes/)
Tools (MCP or built-in)MCP integrations + executor tool lists
Provider configurationRuntime YAML (runtimes/claude.yaml, etc.)
Single-stage executionbare flow -- single stage, one agent
Multi-step recipesdefault flow or custom flow with depends_on
Local executionLocal compute provider (the default)

A one-shot Goose recipe "read the README and add a CONTRIBUTING.md" becomes this Ark recipe:

name: add-contributing
flow: bare
agent: documenter
summary: "Read README and write a CONTRIBUTING.md file"
compute: local

Dispatch with ark session start --recipe add-contributing --repo . --dispatch. You get the same "run a coding agent on my local machine" behavior, plus session tracking, cost recording, and a worktree you can preview with ark worktree diff.

When to use Ark vs Goose

Use Goose if

  • You are a single developer running one agent at a time.
  • Everything is local. You do not need remote compute.
  • You do not need multi-stage SDLC flows (plan -> audit -> implement -> review).
  • You do not need cost tracking across many models or users.
  • You are comfortable with a Python dependency tree.
  • You want something minimal and easy to hack on.

Use Ark if

  • You want to run many sessions in parallel.
  • You have multi-stage SDLC flows: plan, audit, implement, verify, review, document.
  • You need to run agents on remote compute (EC2, K8s, Firecracker, E2B).
  • You are a team or an enterprise hosting agents for many users.
  • You need multi-tenant isolation, API keys, per-tenant policies, and cost caps.
  • You want universal cost tracking across Claude, Codex, Gemini, and subscription plans.
  • You want a knowledge graph that spans code + sessions + memories.
  • You want a TUI, a Web dashboard, and a Desktop app for the same control plane.
  • You prefer a Bun runtime with no Python dependency.

Goose interop (future)

Ark's cli-agent executor can run any CLI tool as a subprocess inside a tmux session with worktree isolation and MCP channel wiring. That means Ark could theoretically drive Goose as one of its runtimes, treating each Goose invocation as an Ark session with all the control-plane benefits wrapped around it (cost tracking, worktree, auto-PR, fan-out, knowledge injection).

This is not currently wired -- we have not built a runtimes/goose.yaml entry -- but the executor and runtime system are designed for exactly this. If you want to use Goose recipes alongside Ark flows, file an issue and we will add a builtin runtime for it.

Honest caveats

  • Ark is bigger and more complex than Goose. If you do not need the orchestration layer, it is overkill.
  • Ark's hosted mode (Postgres, Redis, Helm chart) is written but not fully integration-tested end-to-end yet. See the roadmap for current status.
  • Ark requires tmux and Bun. Goose is pip-installable. If your environment does not have tmux, Ark will not run.
  • Goose has had more time in the wild and its Python runtime may be more familiar to some teams.

Summary

Pick the tool that matches your scale. Goose is a good single-user coding agent runner. Ark is a platform for orchestrating many agents across many compute targets with shared state. If you start with Goose and outgrow it, Ark's concepts are a superset -- your mental model transfers over, and the bare flow gives you a one-stage "run an agent" baseline that looks a lot like a Goose recipe.
