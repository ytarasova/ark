How-To: Knowledge Graph and Codebase Indexing

Index your codebase and let agents query it.

What is the knowledge graph?

The knowledge graph is a unified store of everything Ark knows about your project and its history. It is a single SQLite-or-Postgres structure with typed nodes and typed edges. Every node has a type, a label, optional content, metadata, and a tenant scope. Every edge has a source node, a destination node, and a relationship type.

Node types currently used:

  • file -- a source file in the repo.
  • symbol -- a function, class, method, or type declaration.
  • session -- a historical Ark session with summary and outcome.
  • memory -- user- or agent-written note (see Memories).
  • learning -- a lesson extracted from a retrospective.
  • skill -- a reusable prompt fragment.
  • recipe -- a session template.
  • agent -- an agent role definition.

Edge types include imports, calls, contains, depends_on, co_changes, modified_by, and more. Together they form a graph that any agent can traverse through MCP tools.

This replaces the old pile of standalone systems (memory.ts, learnings.ts, hybrid-search.ts, knowledge.ts) with one unified graph. If you read older docs, ignore them.

Indexing your codebase

The indexer is @optave/codegraph -- an npm package written in TypeScript around a native Rust engine with tree-sitter bindings for 33 languages (WASM grammars). There is no Python dependency. This replaces the earlier Axon-based indexer.

Full index

$ cd ~/my-repo
$ ark knowledge index

This runs codegraph build under the hood, produces .codegraph/graph.db, reads it, and ingests the nodes and edges into Ark's tenant-scoped knowledge and knowledge_edges tables in ~/.ark/ark.db.

Incremental re-index

$ ark knowledge index --incremental

Only re-processes files that changed since the last index. Much faster than a full rebuild on a large codebase.

Automatic indexing

Enable auto-index in ~/.ark/config.yaml:

knowledge:
  auto_index: true
  incremental_index: true
  • Local mode -- honors this config. On first dispatch per repo, runs a full index. On subsequent dispatches, runs incremental. Turn it off if you prefer to index manually.
  • Remote compute (arkd) -- always indexes via the arkd /codegraph/index endpoint, regardless of config. The control plane needs centralized knowledge for all workers, so this behavior is not opt-out.

On a huge codebase the first index can take minutes. Consider running ark knowledge index manually once before your first dispatch so the auto-index step is fast.

Querying from the CLI

Search

$ ark knowledge search "auth"
→ file: src/auth/index.ts
→ symbol: refreshAuthToken (src/auth/refresh.ts)
→ session: "Fix auth token refresh bug" (completed 2026-03-20)
→ memory: Auth token refresh logic is in src/auth/refresh.ts...

Stats

$ ark knowledge stats

Context around a target

$ ark knowledge context --file src/auth.ts

Shows related files (imports, imported by), contained symbols, past sessions that touched the file, and any memories or learnings attached.

Agent MCP tools (6 tools)

Agents query the knowledge graph at runtime through six MCP tools:

ToolPurpose
knowledge/searchFuzzy text search across all node types.
knowledge/contextGet related nodes around a target (imports, calls, co-changes).
knowledge/impact"What breaks if I change this file?" -- traverses reverse dependency edges.
knowledge/historyPast sessions that modified a given file, with their summaries and outcomes.
knowledge/rememberWrite a memory. See Memories.
knowledge/recallRead memories matching a query.

Example transcript of an agent using knowledge/search mid-session:

assistant: I need to understand how auth tokens are refreshed.
           Let me search the knowledge graph first.

tool_use: knowledge/search("auth token refresh")

tool_result:
  - file: src/auth/refresh.ts (imports: jwt, crypto)
  - symbol: refreshAuthToken (defined in src/auth/refresh.ts:42)
  - session s-0x1a3: "Fix expired token retry" (completed 2026-02-11)
  - memory: "Auth token refresh logic is in src/auth/refresh.ts, not auth/index.ts"

assistant: Good -- there is a memory telling me to look in src/auth/refresh.ts,
           and I can see the main function is refreshAuthToken at line 42.
           Let me read that file.

Context injection at dispatch

Before an agent's first turn, Ark builds a knowledge context block and prepends it to the system prompt. The context builder (packages/core/knowledge/context.ts) does this:

  1. Queries the graph for nodes relevant to the session's summary and ticket.
  2. Ranks results by a combination of text relevance, importance (for memories), and recency.
  3. Packs the top N hits into the prompt until the token budget (~2000 tokens) is reached.
  4. Wraps the block in delimiters so the agent knows it's context and not instruction.

This means the agent starts with project awareness. It does not need to blindly grep the codebase on its first turn -- the most relevant files, symbols, past sessions, and memories are already in front of it.

Indexer prerequisite

Install @optave/codegraph either globally or as a repo dependency:

# Option A: per-repo (recommended)
$ bun add @optave/codegraph

# Option B: global
$ npm install -g @optave/codegraph

The binary lives at node_modules/.bin/codegraph (per-repo) or on $PATH (global). Ark finds it automatically.

Supported languages via tree-sitter grammars: TypeScript, JavaScript, Python, Go, Rust, Java, Kotlin, Swift, C, C++, C#, Ruby, PHP, Elixir, Erlang, Haskell, OCaml, Scala, Clojure, Lua, Bash, SQL, YAML, TOML, JSON, HTML, CSS, Vue, Svelte, Markdown, GraphQL, Protobuf, and a few more -- 33 in total.

Worked example

Index a repo, dispatch a session, and observe the knowledge context flowing into the agent's prompt.

# 1. Install the indexer if you haven't
$ cd ~/my-repo
$ bun add @optave/codegraph

# 2. Build the index
$ ark knowledge index
indexing ~/my-repo...
  parsed 2104 files
  extracted 12411 symbols
  computed 2244 co-change edges
wrote 14823 nodes, 41889 edges to ~/.ark/ark.db

# 3. Sanity-check
$ ark knowledge stats

# 4. Dispatch a session that benefits from context
$ ark session start --repo . \
    --summary "Refactor auth module for token rotation" \
    --dispatch --attach

# The agent's system prompt now contains the top ~2000 tokens
# of relevant context: auth files, related symbols, past sessions,
# and any memories you wrote about auth.

Co-change analysis

Ark augments the raw codegraph output with git history. For every pair of files that frequently change in the same commit, Ark writes a co_changes edge with a weight equal to the number of shared commits. This captures implicit coupling that the static graph cannot see -- for example, a test file and its target, or two services in a monorepo that move in lockstep.

Co-change edges power the knowledge/impact MCP tool and give a more accurate "blast radius" than dependency edges alone. If an agent touches a file with heavy co-change weights, it knows to look at the coupled files before calling the work done.

Remote indexing via ArkD

When compute is remote (EC2, K8s pod, Firecracker VM, etc.), the repo lives on the remote target. The flow is:

  1. Conductor POSTs to arkd's /codegraph/index endpoint with { repoPath, incremental }.
  2. ArkD on the remote target runs codegraph build against repoPath.
  3. ArkD parses .codegraph/graph.db on the remote target.
  4. ArkD streams nodes and edges back to the conductor over HTTP.
  5. The conductor writes them into the control-plane knowledge store, tenant-scoped to the session's tenant.

This is why remote indexing is not opt-out in hosted mode -- without the arkd round-trip, the control plane would have no way to build the graph, and agents running on remote compute would lose the knowledge context benefit.

Export and import

You can export the non-codebase nodes (memories, learnings) to Markdown:

$ ark knowledge export --type memory > memories.md
$ ark knowledge export --type learning > learnings.md

And import them back:

$ ark knowledge import memories.md

Code nodes (file, symbol) are not exported -- those come from the indexer and are rebuildable.
