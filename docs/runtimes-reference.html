Runtimes Reference

Runtimes define HOW an agent runs -- the LLM backend, CLI command, billing mode, and transcript parser.

Overview

A runtime is separate from an agent. Agents define WHAT to do (role, prompt, skills). Runtimes define HOW to execute (CLI binary, model catalog, billing). This separation lets you run the same agent on different backends -- e.g. run your implementer agent on Claude Code, Codex, or Gemini without changing the agent definition.

Runtimes use three-tier resolution (highest priority first):

  1. Project: .ark/runtimes/ in your repo
  2. Global: ~/.ark/runtimes/
  3. Builtin: runtimes/ shipped with Ark

Runtime YAML Schema

Create a custom runtime at ~/.ark/runtimes/<name>.yaml or .ark/runtimes/<name>.yaml:

name: my-runtime
description: "Custom LLM backend"
type: cli-agent       # claude-code | cli-agent | subprocess | goose
command: ["my-tool", "--auto"]
task_delivery: arg    # stdin | file | arg
models:
  - id: default
    label: "Default Model"
  - id: fast
    label: "Fast Model"
default_model: default
billing:
  mode: api             # api | subscription | free
  plan: my-plan         # optional, subscription only
  cost_per_month: 200   # optional, subscription only
transcript_parser: claude  # claude | codex | gemini
env:
  MY_VAR: "value"

Executor Types

The runtime's type field selects which executor launches the agent:

claude-code

The default executor for Claude Code. Manages tmux sessions, hooks configuration, MCP channel wiring, trust settings, and worktree isolation.

cli-agent

Generic CLI tool wrapper. Works with any command-line agent (Codex, Gemini CLI, custom tools). Supports three task delivery methods: pass the task as a CLI argument (arg), pipe it via stdin (stdin), or write it to a temp file (file).

subprocess

Direct subprocess spawning. Useful for non-interactive tools like linters, formatters, or build scripts.

goose

Native Goose runtime with recipe dispatch (--recipe / --sub-recipe / --params), MCP channel via --with-extension, router-injected ANTHROPIC_BASE_URL / OPENAI_BASE_URL, stateless mode (--no-session), and stream-JSON output.

Transcript Parsers

Each runtime specifies a transcript parser that reads the agent's conversation history from its native log format. Parsers are registered in TranscriptParserRegistry on AppContext.

ParserSource PathSession Matching
claude~/.claude/projects/<slug>/<session>.jsonlExact path via session.claude_session_id
codex~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonlMatches session_meta.cwd to session.workdir
gemini~/.gemini/tmp/<slug>/chats/session-*.jsonlMatches projectHash = sha256(workdir)

Overriding Runtimes at Dispatch

You can override an agent's default runtime at dispatch time using --runtime:

# Use default runtime (claude) for implementer agent
$ ark session start --repo . --summary "Fix bug" --agent implementer --dispatch

# Override: run implementer on codex runtime
$ ark session start --repo . --summary "Fix bug" --agent implementer --runtime codex --dispatch

# Override: run worker on gemini runtime
$ ark session start --repo . --summary "Fix bug" --agent worker --runtime gemini --dispatch

At dispatch, runtime config (type, command, task_delivery, env, billing) is merged with agent config. Agent-level values take precedence over runtime defaults.

Router URL Injection

When the LLM Router is enabled (router.enabled: true in config), executors automatically inject ANTHROPIC_BASE_URL and OPENAI_BASE_URL into the agent's environment at dispatch. This routes all LLM calls through the router transparently -- the agent binary has no awareness of the routing layer.

The injection flow: agent → Router (:8430) → TensorZero (:3000, optional) → Provider API.

CLI Commands

# List all available runtimes
$ ark runtime list

# Show details of a specific runtime
$ ark runtime show claude

# Show details of the goose runtime
$ ark runtime show goose

Plugin Executors

You can extend Ark with custom executor plugins. Drop a JavaScript module at ~/.ark/plugins/executors/<name>.js that exports an Executor interface. Ark discovers and registers plugin executors at boot.

The executor interface requires 5 methods:

  • launch(opts) -- start an agent and return a handle
  • kill(handle) -- stop a running agent
  • status(handle) -- query agent state (running, idle, completed, failed, not_found)
  • send(handle, message) -- send a message to a running agent
  • capture(handle, lines?) -- capture recent output from the agent

Plugin failures are silently skipped -- a broken plugin never blocks Ark boot.
