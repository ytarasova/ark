LLM Router

OpenAI-compatible proxy that routes requests across multiple LLM providers with policy-based selection, circuit breakers, and cost tracking.

Overview

The LLM Router sits between your agents and LLM providers. It exposes an OpenAI-compatible /v1/chat/completions endpoint and intelligently routes each request to the best model based on your configured policy. When the router is enabled, executors automatically inject ANTHROPIC_BASE_URL and OPENAI_BASE_URL into agent environments so all LLM traffic flows through the router transparently.

Source: packages/router/. Default port: 8430.

Quick Start

# Start the router with default balanced policy
$ ark router start

# Start with a specific policy and port
$ ark router start --policy quality --port 8430

# Check router status
$ ark router status

# View accumulated costs
$ ark router costs

Routing Policies

The router supports three routing policies that control how it selects a model for each request:

quality

Always selects the highest-quality model available. No cost optimization. Best for critical tasks where output quality matters most.

balanced

Selects the cheapest model that meets a quality floor (default: 0.8). Upgrades to higher-quality models when the request requires tools or complex reasoning. This is the default policy and a good starting point for most workloads.

cost

Minimizes cost aggressively. Selects the cheapest available model. Automatically upgrades when the request contains tool calls (since cheap models often struggle with tools). Best for high-volume, simple tasks.

Request Classification

Before routing, the classifier analyzes each request to determine its complexity. This informs the routing decision.

SignalWhat it measures
Message lengthLonger prompts indicate more complex tasks
Turn countMulti-turn conversations may need stronger models
Tool presenceRequests with tools need models that support them well
Code indicatorsCode generation and analysis tasks
Reasoning indicatorsTasks requiring logical reasoning or planning

The classifier outputs a complexity score (0-1), task type (code, reasoning, extraction, generation, chat), and difficulty level (trivial, simple, moderate, complex, expert).

Model Pool

The router maintains a pool of available models organized by tier:

TierModels
FrontierClaude Opus, GPT-4.1
StandardClaude Sonnet, GPT-4.1-mini, Gemini 2.5 Pro
EconomyClaude Haiku, GPT-4.1-nano, Gemini 2.5 Flash

Configure the model pool in ~/.ark/config.yaml under router.models. Each model entry specifies the provider, model ID, quality score, cost tier, and supported features (streaming, tools, vision).

Circuit Breakers

The router includes per-provider circuit breakers that prevent cascading failures:

  • Closed (normal): requests flow through. Failures are counted.
  • Open (tripped): after 5 consecutive failures, the circuit opens. All requests to this provider fail immediately and are routed to alternatives.
  • Half-open (recovery): after a 30-second cooldown, one test request is allowed through. If it succeeds, the circuit closes. If it fails, it re-opens.

Circuit breakers ensure that a down provider does not block all agent work. The router automatically falls back to same-tier alternatives.

Sticky Sessions

For multi-turn conversations, the router uses sticky sessions to maintain model continuity. Once a model is selected for a session, subsequent requests in the same session go to the same model (or an equivalent). The session sticks for 1 hour (TTL: 3,600,000ms).

The router can also escalate within a sticky session -- if a conversation becomes more complex (e.g. the agent starts using tools), the router may upgrade to a higher-tier model mid-session.

Cascade Mode

In cascade mode, the router tries the cheapest model first. If the response quality is insufficient (based on heuristics), it re-routes to a higher-tier model. This provides aggressive cost optimization while maintaining a quality floor.

Cost Tracking

The router tracks per-request costs and feeds them into Ark's UsageRecorder via an onUsage callback. Costs are attributed by model, provider, and session. View accumulated costs with:

$ ark router costs

The cost dashboard is also available in the Web UI (Costs view).

Feedback Loop

The router logs every routing decision for analysis. Per-model quality tracking uses Bayesian scoring to refine model selection over time. The feedback system maintains the last 10,000 decision records.

TensorZero Integration

When TensorZero is enabled, the router routes matched requests through the TensorZero gateway instead of calling provider APIs directly. TensorZero is an open-source Rust LLM gateway (Apache 2.0) that provides additional observability and experiment management.

Configuration

# ~/.ark/config.yaml
tensorZero:
  enabled: true
  port: 3000
  config_dir: ~/.ark/tensorzero
  auto_start: true

Startup Sequence

TensorZero starts in one of three modes (tried in order):

  1. Sidecar detect -- reuse an already-running TensorZero instance on the configured port.
  2. Native binary -- spawn tensorzero-gateway if installed (e.g. next to the ark binary).
  3. Docker fallback -- run the official tensorzero/gateway Docker image.

Config (tensorzero.toml) is auto-generated from your configured provider API keys at boot.

Traffic Flow

With TensorZero enabled, the full request path is:

agent → Router (:8430) → TensorZero (:3000) → Provider API

Without TensorZero, the router calls provider APIs directly:

agent → Router (:8430) → Provider API

Configuration Reference

# ~/.ark/config.yaml
router:
  enabled: true
  port: 8430
  policy: balanced          # quality | balanced | cost
  quality_floor: 0.8       # 0-1, used by balanced policy
  auto_start: true
  models:
    - provider: anthropic
      model: claude-opus-4-6
      tier: frontier
    - provider: anthropic
      model: claude-sonnet-4-6
      tier: standard
    - provider: openai
      model: gpt-4.1
      tier: frontier
    # ... more models

Tenant Policy Integration

In hosted mode, tenant policies can control router behavior:

PolicyEffect
router_enabledWhether the tenant can use the router
router_requiredForce all traffic through the router (overrides agent opt-out)
router_policyOverride routing policy for the tenant (quality/balanced/cost)
tensorzero_enabledWhether TensorZero is available for this tenant

Use getEffectiveIntegrationSettings() to merge tenant policy with global config -- tenant _required flags override global opt-outs.
