# Awilix DI Container Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace manual DI and `getApp()` calls with an awilix container. AppContext becomes a thin wrapper around the container. All services resolve through awilix with proper lifecycle (boot/shutdown in correct order, scoped containers for tests and multi-tenancy).

**Architecture:** Single awilix container owns all registrations. Services declare their dependencies via constructor parameters (CLASSIC injection mode). Tests get scoped containers with isolated DBs. Multi-tenancy (future) uses `createScope()` per tenant.

**Tech Stack:** awilix 13.x, TypeScript, Bun

### Task 1: Define the container and cradle type

**Create:** `packages/core/container.ts`

This file defines what's in the container (the "cradle") and how to build it.

```ts
// packages/core/container.ts
import {
  createContainer,
  asClass,
  asValue,
  asFunction,
  InjectionMode,
  type AwilixContainer,
} from "awilix";
import type { ArkConfig } from "./config.js";
import type { SessionRepository } from "./repositories/session.js";
import type { ComputeRepository } from "./repositories/compute.js";
import type { EventRepository } from "./repositories/event.js";
import type { MessageRepository } from "./repositories/message.js";
import type { TodoRepository } from "./repositories/todo.js";
import type { SessionService } from "./services/session.js";
import type { ComputeService } from "./services/compute.js";
import type { HistoryService } from "./services/history.js";
import type { FlowStore } from "./stores/flow-store.js";
import type { SkillStore } from "./stores/skill-store.js";
import type { AgentStore } from "./stores/agent-store.js";
import type { RecipeStore } from "./stores/recipe-store.js";

/**
 * The cradle -- everything that can be resolved from the container.
 * This is the single source of truth for all dependencies.
 */
export interface Cradle {
  // Config
  config: ArkConfig;

  // Database
  db: import("bun:sqlite").Database;

  // Repositories
  sessions: SessionRepository;
  computes: ComputeRepository;
  events: EventRepository;
  messages: MessageRepository;
  todos: TodoRepository;

  // Services
  sessionService: SessionService;
  computeService: ComputeService;
  historyService: HistoryService;

  // Resource stores
  flows: FlowStore;
  skills: SkillStore;
  agents: AgentStore;
  recipes: RecipeStore;
}

export type AppContainer = AwilixContainer<Cradle>;

/**
 * Create the DI container with all registrations.
 * Does NOT open DB or start services -- that's boot().
 */
export function createAppContainer(config: ArkConfig): AppContainer {
  const container = createContainer<Cradle>({
    injectionMode: InjectionMode.CLASSIC,
    strict: true,
  });

  container.register({ config: asValue(config) });

  return container;
}
```

### Task 2: Rewrite AppContext.boot() to use the container

**Modify:** `packages/core/app.ts`

Replace the manual `this._sessions = new SessionRepository(this._db)` pattern with awilix registrations. The container owns all instances and their lifecycle.

```ts
// In boot():
import { asClass, asValue, asFunction } from "awilix";

// 1. Open database
const db = new Database(this.config.dbPath);
db.run("PRAGMA journal_mode = WAL");
db.run("PRAGMA busy_timeout = 5000");
initRepoSchema(db);
seedLocalCompute(db);

this.container.register({
  db: asValue(db),

  // Repositories (singletons -- one per container lifetime)
  sessions: asClass(SessionRepository).singleton(),
  computes: asClass(ComputeRepository).singleton(),
  events: asClass(EventRepository).singleton(),
  messages: asClass(MessageRepository).singleton(),
  todos: asClass(TodoRepository).singleton(),

  // Services
  sessionService: asClass(SessionService).singleton(),
  computeService: asClass(ComputeService).singleton(),
  historyService: asClass(HistoryService).singleton(),

  // Resource stores
  flows: asValue(new FileFlowStore({ builtinDir, userDir: ... })),
  skills: asValue(new FileSkillStore({ ... })),
  agents: asValue(new FileAgentStore({ ... })),
  recipes: asValue(new FileRecipeStore({ ... })),
});
```

**Accessors** become container resolves:
```ts
get sessions(): SessionRepository { return this.container.resolve("sessions"); }
get sessionService(): SessionService { return this.container.resolve("sessionService"); }
// etc.
```

**shutdown()** disposes the container:
```ts
async shutdown() {
  // Stop sessions first (needs services + providers)
  if (this.opts?.cleanupOnShutdown) {
    await this.container.resolve("sessionService").stopAll();
  }
  // Dispose container (closes DB, cleans up)
  await this.container.dispose();
}
```

### Task 3: Update repository constructors for awilix resolution

Awilix CLASSIC mode injects by constructor parameter NAME. So repositories need their constructor param named `db`:

```ts
// Already correct -- SessionRepository(db: Database) matches cradle.db
export class SessionRepository {
  constructor(private db: Database) {}
}
```

All repos already take `db` as first param -- this should just work.

### Task 4: Update service constructors for awilix resolution

Services need their constructor params to match cradle keys:

```ts
// packages/core/services/session.ts
export class SessionService {
  constructor(
    private sessions: SessionRepository,   // matches cradle.sessions
    private events: EventRepository,       // matches cradle.events
    private messages: MessageRepository,   // matches cradle.messages
  ) {}
}
```

**IMPORTANT:** Remove the `app: AppContext` parameter we added earlier. Services should depend on specific repos/services, NOT the entire container. If a service needs to call orchestration functions, it resolves them through the container.

For orchestration functions that need `app`, we have two options:
- A) Pass the container/app when calling (current approach)
- B) Register orchestration functions in the container too

Go with A for now -- orchestration is functional, not class-based.

### Task 5: AppContext.forTest() creates a scoped container

```ts
static forTest(overrides?: Partial<ArkConfig>): AppContext {
  const tempDir = mkdtempSync(join(tmpdir(), "ark-test-"));
  const config = loadConfig({ arkDir: tempDir, env: "test", ...overrides });
  const app = new AppContext(config, {
    skipConductor: true,
    skipMetrics: true,
    skipSignals: true,
    cleanupOnShutdown: true,
  });
  return app;
}
```

Tests use `AppContext.forTest()` which creates a fresh container with an isolated DB. On `shutdown()`, the container is disposed, closing the DB and cleaning up.

## Dependency Graph (boot order)

```
config (value)
  └─> db (value, depends on config.dbPath)
       ├─> sessions (class, depends on db)
       ├─> computes (class, depends on db)
       ├─> events (class, depends on db)
       ├─> messages (class, depends on db)
       └─> todos (class, depends on db)
            ├─> sessionService (class, depends on sessions, events, messages)
            ├─> computeService (class, depends on computes)
            └─> historyService (class, depends on db)

flows, skills, agents, recipes (values, depend on config.arkDir)
```

## Shutdown Order (reverse of boot, plus process cleanup)

```
1. sessionService.stopAll() -- kill running agents via providers
2. Stop conductor, metrics, pollers
3. container.dispose() -- disposes all registered disposers:
   - db.close()
   - Any other cleanup registered via .disposer()
4. Clear singleton reference
5. Clean temp dir (test mode)
```

## Key Principle

**Services declare what they need. The container provides it. No service ever calls getApp().**
