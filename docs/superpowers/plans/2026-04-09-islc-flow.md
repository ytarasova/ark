# ISLC Flow: Intent-to-Software Lifecycle in Ark

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Implement the full ISLC pipeline as native Ark flows, agents, and skills -- replacing Goose recipes entirely. A user runs `ark session start --ticket IN-1234 --flow islc --dispatch` and gets: ticket intake -> spec extraction -> planning -> audit -> execute (fan-out) -> verify -> close (PR + Confluence) -> retro.

**Architecture:** Each ISLC stage is an Ark agent with specialized skills. MCP servers provide Jira, Confluence, and Figma access. State passes between stages via `.workflow/<ticket>/` directory in the worktree. Fan-out at execute stage decomposes subtasks into parallel child sessions.

---

## What we need to build

### Layer 1: MCP Servers (data access)

These give agents access to external systems. Agents call them via MCP tools -- no direct HTTP.

**1a. Jira MCP Server** (`mcp-servers/jira/`)
- `jira/getIssue` -- fetch full ticket (summary, description, AC, links, comments)
- `jira/createIssue` -- create sub-tasks linked to parent
- `jira/transition` -- move ticket status (To Do -> In Progress -> Done)
- `jira/search` -- JQL search for duplicates, blockers
- `jira/addComment` -- post PR URLs, status updates
- `jira/getTransitions` -- list available transitions for a ticket
- Auth: API token from env `JIRA_API_TOKEN` + `JIRA_BASE_URL`

**1b. Confluence MCP Server** (`mcp-servers/confluence/`)
- `confluence/createPage` -- publish implementation notes
- `confluence/updatePage` -- append retro findings
- `confluence/search` -- CQL search for related docs
- `confluence/getPage` -- fetch background context
- Auth: same Atlassian token as Jira

**1c. Figma MCP Server** (`mcp-servers/figma/`)
- `figma/getDesignContext` -- component tree, layout, annotations
- `figma/getScreenshot` -- visual reference image
- `figma/getTokens` -- design tokens (colors, spacing, typography)
- Auth: `FIGMA_ACCESS_TOKEN`

### Layer 2: Skills (reusable prompt fragments)

**2a. `skills/spec-extraction.md`**
The 27-section specification extraction prompt. Injected into the ticket-intake agent.

**2b. `skills/sanity-gate.md`**
The 11-field sanity gate checklist. Used by both the ideate recipe and ticket-intake stage.

**2c. `skills/plan-audit.md`**
Coverage matrix generation -- map requirements to subtasks, find gaps.

**2d. `skills/security-scan.md`**
Security review checklist (hardcoded secrets, SQL injection, unvalidated input, CVEs).

**2e. `skills/self-review.md`**
Pre-PR code review checklist (dead code, debug statements, error handling, docs).

### Layer 3: Agents (specialized per stage)

**3a. `agents/ticket-intake.yaml`**
```yaml
name: ticket-intake
description: Fetch Jira ticket, run 11-field sanity gate, extract 27-section spec
model: sonnet
max_turns: 50
skills: [sanity-gate, spec-extraction]
mcp_servers: [jira, figma]
system_prompt: |
  You are the ISLC Ticket Intake agent. Your job:
  1. Fetch the Jira ticket via jira/getIssue
  2. Run the 11-field sanity gate (from your sanity-gate skill)
  3. If FAIL, report blockers and stop
  4. If PASS, extract the 27-section spec (from your spec-extraction skill)
  5. Resolve any Figma URLs via figma/getDesignContext
  6. Write spec.md to {workdir}/.workflow/{ticket}/spec.md
  7. Write state.json with sanity result and spec path
```

**3b. `agents/islc-planner.yaml`**
```yaml
name: islc-planner
description: Decompose spec into ordered subtasks, create Jira sub-tasks
model: opus
max_turns: 100
mcp_servers: [jira]
system_prompt: |
  You are the ISLC Planning agent. Your job:
  1. Read the spec from {workdir}/.workflow/{ticket}/spec.md
  2. Decompose into independently-executable subtasks
  3. Create each as a Jira sub-task via jira/createIssue (parent = {ticket})
  4. Write plan.md with subtask table + dependency graph
  5. Write subtasks.json mapping T-N to Jira keys
  6. Update state.json
```

**3c. `agents/plan-auditor.yaml`**
```yaml
name: plan-auditor
description: Audit plan coverage against spec requirements
model: sonnet
skills: [plan-audit]
mcp_servers: [jira, figma]
system_prompt: |
  You are the ISLC Plan Auditor. Your job:
  1. Extract atomic requirements from spec.md
  2. Map each to subtasks in plan.md
  3. Build coverage matrix (COVERED / PARTIAL / UNCOVERED)
  4. If CRITICAL gaps exist, report FAIL
  5. Write audit-report.md
```

**3d. `agents/islc-implementer.yaml`**
```yaml
name: islc-implementer
description: Implement a single subtask from the ISLC plan
model: opus
max_turns: 200
mcp_servers: [jira, figma]
system_prompt: |
  You are implementing subtask {summary} for ticket {ticket}.
  1. Read the subtask definition from plan.md
  2. Transition Jira sub-task to In Progress via jira/transition
  3. If frontend task, fetch Figma context via figma/getDesignContext
  4. Write code, tests, commit
  5. Transition Jira sub-task to Done
  6. Update subtasks.json status
```

**3e. `agents/islc-verifier.yaml`**
```yaml
name: islc-verifier
description: Verify implementation against spec, tests, security, AC
model: sonnet
skills: [security-scan, self-review]
mcp_servers: [jira, figma]
system_prompt: |
  You are the ISLC Verifier. Run 6 verification layers:
  1. Automated tests (run test suite)
  2. Security scan (from security-scan skill)
  3. Code quality review (from self-review skill)
  4. AC validation (map each AC to test/code evidence)
  5. Design UAT (compare with Figma via figma/getScreenshot)
  6. Write verify-report.md with PASS/FAIL verdict
```

**3f. `agents/islc-closer.yaml`**
```yaml
name: islc-closer
description: Create PR, transition Jira, publish Confluence
model: sonnet
mcp_servers: [jira, confluence]
system_prompt: |
  You are the ISLC Closer. Your job:
  1. Verify all sub-tasks Done in Jira
  2. Create PR (via git push + gh pr create)
  3. Transition parent Jira ticket to In Review
  4. Add PR link as Jira comment
  5. Create Confluence implementation page
  6. Update state.json with PR + Confluence URLs
```

**3g. `agents/islc-retro.yaml`**
```yaml
name: islc-retro
description: Automated retrospective on the ISLC workflow run
model: sonnet
mcp_servers: [jira, confluence]
system_prompt: |
  You are the ISLC Retrospective agent. Analyse the completed workflow:
  1. Reconstruct timeline from state.json timestamps
  2. Score spec quality, plan quality, execution quality
  3. Generate 3-7 improvement recommendations
  4. Write retro-report.md
  5. Optionally append to Confluence page
```

### Layer 4: Flow Definition

**4a. `flows/definitions/islc.yaml`**
```yaml
name: islc
description: "Intent-to-Software Lifecycle -- full pipeline from Jira ticket to merged PR"
stages:
  - name: ticket-intake
    agent: ticket-intake
    gate: auto
    task: |
      Process Jira ticket {ticket}. Run sanity gate and extract 27-section spec.
      Write artifacts to {workdir}/.workflow/{ticket}/

  - name: plan
    agent: islc-planner
    gate: manual
    depends_on: [ticket-intake]
    task: |
      Read spec from {workdir}/.workflow/{ticket}/spec.md.
      Decompose into subtasks, create Jira sub-tasks.

  - name: audit
    agent: plan-auditor
    gate: auto
    depends_on: [plan]
    optional: true
    task: |
      Audit plan coverage against spec. Write audit-report.md.

  - name: execute
    type: fan_out
    agent: islc-implementer
    gate: auto
    depends_on: [audit]
    max_parallel: 4
    task: |
      Execute subtasks from plan. Each child implements one subtask.

  - name: verify
    agent: islc-verifier
    gate: manual
    depends_on: [execute]
    verify:
      - "npm test"
      - "npm run lint"
    task: |
      Run verification layers. Write verify-report.md.

  - name: close
    agent: islc-closer
    gate: auto
    depends_on: [verify]
    task: |
      Create PR, transition Jira, publish Confluence.

  - name: retro
    agent: islc-retro
    gate: auto
    depends_on: [close]
    optional: true
    task: |
      Analyse workflow run. Write retro-report.md.
```

**4b. `flows/definitions/islc-quick.yaml`** (skip audit + retro)
```yaml
name: islc-quick
description: "Fast ISLC -- ticket intake -> plan -> execute -> verify -> close"
stages:
  - name: ticket-intake
    agent: ticket-intake
    gate: auto
  - name: plan
    agent: islc-planner
    gate: manual
    depends_on: [ticket-intake]
  - name: execute
    type: fan_out
    agent: islc-implementer
    gate: auto
    depends_on: [plan]
  - name: verify
    agent: islc-verifier
    gate: auto
    depends_on: [execute]
  - name: close
    agent: islc-closer
    gate: auto
    depends_on: [verify]
```

    description: "Jira project key"

### Layer 6: Workflow State

Stage agents read/write state to `.workflow/<ticket>/` in the session worktree:

```
.workflow/IN-1234/
  state.json       -- stage progress, timestamps, results
  spec.md          -- 27-section specification
  plan.md          -- subtask table + dependency graph
  subtasks.json    -- T-N -> Jira key mapping
  audit-report.md  -- coverage matrix
  verify-report.md -- test/security/AC results
  retro-report.md  -- retrospective findings
  tasks/
    IN-1235/
      PLAN.md
      SUMMARY.md
      VERIFICATION.md
```

This is local to the worktree. Each session gets its own copy.

## Usage

```bash
# Full ISLC pipeline
ark session start --ticket IN-1234 --flow islc --dispatch

# Quick ISLC (skip audit + retro)
ark session start --ticket IN-1234 --flow islc-quick --dispatch

# From recipe
ark session start --recipe islc --ticket IN-1234 --dispatch

# PM ideation
ark session start --recipe ideate --idea "Add OAuth2 login" --project-key IN --dispatch

# Resume from a specific stage
ark session start --ticket IN-1234 --flow islc --from-stage execute --dispatch
```

10. Ensure `.workflow/` directory is created in worktree at dispatch
