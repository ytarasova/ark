How-To: Working with Memories

Store and recall knowledge across sessions.

What are memories?

Memories are persistent notes that survive across sessions. Once written, a memory becomes part of the knowledge graph and can be referenced by any agent on any future dispatch in the same scope. They are how you teach Ark things you do not want to repeat in every prompt -- coding conventions, architectural decisions, project-specific gotchas, debugging shortcuts.

Internally, a memory is a memory:* node in the knowledge table (tenant-scoped). Each memory has:

  • content -- the free-form text you want to remember.
  • importance -- a float between 0.0 and 1.0. Higher-importance memories win the context-budget lottery at dispatch time.
  • scope -- global (available to every repo) or project (tied to one repo path).
  • created_at -- timestamp of when the memory was written.
  • metadata -- freeform JSON blob for tagging or extra fields.

The old ~/.ark/memories.json and ~/.ark/LEARNINGS.md files no longer exist. Everything lives in the knowledge graph alongside codebase structure, session history, skills, and recipes.

Three ways to add memories

1. From an agent, via MCP

While an agent is running, it can call the knowledge/remember MCP tool. This is the main way agents accumulate long-lived learning. For example, during a session an agent might write:

knowledge/remember({
  content: "Auth token refresh logic is in src/auth/refresh.ts, not auth/index.ts",
  importance: 0.85,
  scope: "project"
})

The memory is written to the graph immediately and becomes available to the next session in the same repo.

2. From the CLI

Write a memory manually with ark knowledge remember:

$ ark knowledge remember "Always prefer functional React components over class components" \
    --importance 0.9 --scope global

This is useful for recording conventions before you start a new project, or for dropping in notes you have in an old README.

3. Importing from Markdown

If you have existing docs, design notes, or architecture decisions in Markdown, import them in one shot:

$ ark knowledge import notes.md

Import/export round-trips with YAML front matter so you can version your memories in git if you want to.

Recall and search

Memories are searchable like any other knowledge node.

Recall by fuzzy query

$ ark knowledge recall "auth token"
→ memory: Auth token refresh logic is in src/auth/refresh.ts... (importance 0.85)
→ memory: Token rotation schedule lives in ops/rotate.yaml (importance 0.6)

Full-text search

$ ark knowledge search "auth token"

This runs against the FTS5 index and returns every matching knowledge node, not just memories.

Automatic injection at dispatch

When you dispatch a session, Ark queries the knowledge graph for nodes relevant to the task summary and injects the top hits into the agent's system prompt. Memories participate automatically, ranked by importance and relevance. The injection is token-budgeted to roughly 2000 tokens so a very large knowledge base will not blow out the context window.

The agent does not need to ask for memories -- they arrive as part of its initial context, tagged so it knows they came from the graph.

Export to Markdown

Export all memories to a Markdown file:

$ ark knowledge export --type memory > memories.md

Each memory becomes a Markdown section with YAML front matter:

---
type: memory
importance: 0.9
scope: global
created_at: 2026-04-01T14:22:10Z
---

Always prefer functional React components over class components.

You can edit the file and re-import it. The import is idempotent -- memories with the same content and scope are deduped.

Scopes

Memories come in two scopes:

  • global -- applies to every session you run, regardless of repo. Good for language-level conventions, personal style preferences, or universal "do not do X" rules.
  • project -- tied to a specific repository. Good for architectural notes, "this weird thing exists for reason Y", or hard-won debugging shortcuts specific to a codebase.

Project scope is inferred from the session's repo field at dispatch time. If an agent writes a memory with scope: project, it is attached to the current repo and will only surface in future sessions in that same repo.

Worked example

Let's teach Ark a convention, then verify the convention gets into the agent's prompt on the next dispatch.

# 1. Write a memory from the CLI
$ ark knowledge remember \
    "Always prefer functional React components over class components" \
    --importance 0.9 --scope global

# 2. Verify it landed in the graph
$ ark knowledge recall "functional components"

# 3. Dispatch a new session
$ ark session start --repo . \
    --summary "Add user profile page" --dispatch

# 4. Attach and inspect the agent's system prompt
$ ark session attach <session-id>
# The memory is injected as part of the knowledge context block

The agent will see the memory as part of its initial context and should naturally honor the convention when it writes the profile page component.

Editing and deleting

Memories are plain rows in the knowledge table. To replace one, write a new memory with the same content (the import step will dedupe) or rewrite it via a Markdown export/import cycle. To delete, remove it from the exported file and re-import with --replace, or drop the row directly via SQL in ~/.ark/ark.db.

Relationship to learnings

"Learnings" are a special memory type -- notes an agent writes as post-mortems after a session. They go through the same knowledge/remember path but carry a learning tag in metadata so the retro agent can surface them during reviews. For the casual user, memories and learnings are the same thing -- just different tags on the same graph nodes.

Multi-tenant note

In hosted mode, memories are automatically tenant-scoped. The knowledge and knowledge_edges tables carry a tenant_id column, and every read/write goes through the tenant-scoped AppContext view. That means tenant A's memories are invisible to tenant B -- there is no global memory pool in hosted mode.

Per-user memories within a tenant are not currently separated: if two users on the same tenant write memories to the same project scope, they will both see each other's memories. This is intentional for team knowledge sharing, but worth knowing when onboarding new team members.

Command reference

CommandWhat it does
ark knowledge remember "text"Write a new memory. Flags: --importance, --scope.
ark knowledge recall <query>Fuzzy search memories only.
ark knowledge search <query>Full-text search across all knowledge nodes.
