Architecture

How Ark's components fit together.

Overview

Ark runs in two deployment modes from a single codebase:

  • Local single-user mode -- SQLite, file-backed stores, no auth, everything in ~/.ark/.
  • Hosted / control plane mode -- PostgreSQL, DB-backed stores, API-key auth, worker registry, tenant policies, scheduler.

The mode is determined by config: set DATABASE_URL to a Postgres connection string to enable hosted mode. See Multi-Tenancy for deployment details.

Core components

  • AppContext -- central root, built on an Awilix DI container. Wires repositories, services, stores, knowledge, conductor, and config. Created at CLI startup (without conductor) or server daemon startup (with conductor).
  • DatabaseAdapter abstraction -- database.ts defines the interface, database-sqlite.ts implements it for bun:sqlite (local), database-postgres.ts implements it for PostgreSQL (hosted). All repositories and stores use DatabaseAdapter, never raw bun:sqlite.
  • Repositories -- SQL CRUD behind typed classes: SessionRepository, ComputeRepository, ComputeTemplateRepository, EventRepository, MessageRepository, TodoRepository. Column whitelists prevent injection. Accessed via app.sessions, app.computes, etc.
  • Services -- SessionService (lifecycle facade), ComputeService, HistoryService, and session-orchestration.ts for dispatch, advance, fork, clone, spawn, fan-out, handoff, worktree ops, and report handling. Every orchestration function takes app: AppContext as its first argument -- no global state.
  • Stores -- Resource stores with three-tier file resolution (builtin > global > project): FlowStore, SkillStore, AgentStore, RecipeStore, RuntimeStore. Hosted mode swaps in DbResourceStore which reads from the tenant-scoped resource_definitions table.
  • KnowledgeStore -- Unified graph of nodes (files, symbols, sessions, memories, learnings, skills, recipes, agents) and edges (depends_on, imports, modified_by, etc.). Stored in the tenant-scoped knowledge and knowledge_edges tables.
  • PricingRegistry + UsageRecorder -- 300+ model prices via LiteLLM JSON. Every assistant turn produces a usage_records row with cost_mode (api / subscription / free).
  • TranscriptParserRegistry -- Polymorphic per-runtime transcript parsers on AppContext.
  • TenantPolicyManager, ApiKeyManager -- Multi-tenant auth and policy enforcement. Used in hosted mode.

Executors

Agents dispatch through pluggable executors. The runtime type field selects which executor launches the agent:

  • claude-code (default) -- Claude Code in tmux with hooks and an MCP channel.
  • subprocess -- spawns any command as a child process.
  • cli-agent -- runs any CLI tool (Codex, Gemini CLI, etc.) in tmux with worktree isolation.

Executors are registered at boot in app.ts. The registry interface is defined in packages/core/executor.ts with five methods: launch, kill, status, send, capture.

When the router is enabled, executors inject ANTHROPIC_BASE_URL and OPENAI_BASE_URL into the agent's environment at dispatch via the router-env.ts helper. All LLM calls transparently flow Router -> (TensorZero ->) Provider.

Conductor

The conductor is the central HTTP server on port 19100. Responsibilities:

  • Receive hook status reports (busy, idle, error, done) from Claude Code agents.
  • Relay channel reports between agents and humans.
  • Serve the GitHub webhook endpoint for PR review monitoring.

The conductor receives its AppContext at construction via startConductor(app, port) -- no getApp() calls. It delegates hook-status and report handling to session-orchestration.ts. The CLI skips the conductor; the server daemon runs it.

ArkD

ArkD is the universal agent daemon. A stateless HTTP server on port 19300 that runs on every compute target (local, Docker, devcontainer, EC2, K8s pod sidecar, Firecracker micro-VM). It handles agent lifecycle, file ops, exec, metrics, channel relay, and codegraph indexing.

See ArkD Daemon for the full endpoint reference and deployment details.

MCP socket pooling

Problem: Each Ark session runs multiple agents, and each agent has multiple MCP servers (knowledge graph, filesystem, context7, playwright, etc.). Without pooling, N sessions x M MCP servers = N*M processes. Five sessions x six MCP servers = 30 processes. Memory blows up fast.

Solution: A SocketProxy wraps a single MCP process per server name and accepts multi-client connections over a unix domain socket at /tmp/ark-mcp-<name>.sock. Client sessions connect via:

{
  "command": "ark",
  "args": ["mcp-proxy", "/tmp/ark-mcp-<name>.sock"]
}

Health monitoring and auto-restart keep the pooled processes alive. Result: 85-90% memory reduction in practice.

Config (under mcp_pool: in ~/.ark/config.yaml):

mcp_pool:
  enabled: true
  autoStart: true
  poolAll: true
  excludeMcps: ["filesystem"]

Channels

Ark uses the official Claude Code claude/channel protocol for bidirectional agent communication.

  • Inbound: notifications/claude/channel events show up inside the agent as <channel source="ark" ...> tags.
  • Outbound: two MCP tools on the ark-channel server: report (progress, completion, error, question) and send_to_agent (cross-agent messaging for handoffs and fan-out).

Data flow:

  1. Session dispatch writes .mcp.json with an ark-channel entry and a deterministic port 19200 + (hash(sessionId) % 10000).
  2. Claude Code spawns ark-channel over stdio.
  3. Agent reports go via ark-channel -> arkd (HTTP) -> conductor (HTTP).
  4. Human steering from the Web UI or Desktop app goes via conductor -> arkd -> ark-channel -> agent MCP notification.

The reason agents do not talk to the conductor directly is that agents run on potentially remote compute. Going through arkd gives the control plane one HTTP endpoint (port 19300) per compute target, regardless of how many sessions are on it.

Compute providers

Ark supports 11 compute providers across four categories:

  • Local (worktree, no isolation) -- local.
  • Local with isolation -- docker, devcontainer, firecracker.
  • Remote via arkd (EC2 + arkd daemon) -- ec2, ec2-docker, ec2-devcontainer, ec2-firecracker.
  • Managed / cluster -- e2b, k8s, k8s-kata.

See Compute Providers for a full breakdown.

Transcript parsers

Ark parses agent transcripts to produce universal usage records regardless of runtime. TranscriptParserRegistry lives on AppContext and dispatches to polymorphic per-runtime parsers in packages/core/runtimes/<name>/parser.ts.

All three parsers use workdir-based session identification rather than "latest by mtime":

  • ClaudeTranscriptParser -- exact path ~/.claude/projects/<slug>/<session>.jsonl via session.claude_session_id.
  • CodexTranscriptParser -- scans ~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl, matches session_meta.payload.cwd to session.workdir.
  • GeminiTranscriptParser -- scans ~/.gemini/tmp/<slug>/chats/session-*.jsonl, matches projectHash = sha256(workdir) against the file's first line.

Parser output feeds UsageRecorder which writes a row to usage_records with the effective cost_mode.

LLM Router and TensorZero

The LLM Router (packages/router/) is an OpenAI-compatible HTTP proxy with three routing policies (quality, balanced, cost), circuit breakers, request classification, and cost tracking via an onUsage callback wired to UsageRecorder.

TensorZero is an optional Rust LLM gateway (Apache 2.0) that sits behind the router as a unified provider. The lifecycle manager in packages/core/router/tensorzero.ts starts TensorZero in one of three modes, tried in order: sidecar detect (reuse running instance), native binary (spawn tensorzero-gateway), Docker fallback.

Config (tensorzero.toml) is generated from configured provider API keys at boot. When router.auto_start and tensorZero.enabled are both true, TensorZero auto-starts with the router. Flow: agent -> Router -> TensorZero -> Provider.

Executors inject ANTHROPIC_BASE_URL and OPENAI_BASE_URL into the agent's environment at dispatch so the agent binary is unaware anything has changed.

Knowledge graph and ops-codegraph

Ark's knowledge graph is a unified store of codebase structure, session history, memories, and learnings. The indexer is powered by @optave/codegraph -- a Rust engine with tree-sitter WASM bindings for 33 languages. No Python dependency.

The indexer runs codegraph build to produce .codegraph/graph.db, then ingests the nodes and edges into Ark's tenant-scoped knowledge and knowledge_edges tables. It also runs git co-change analysis.

Auto-index on dispatch:

  • Local compute -- honors knowledge.auto_index in ~/.ark/config.yaml.
  • Remote compute -- always indexes via the arkd /codegraph/index endpoint. The control plane needs centralized knowledge for all workers.

Agents query the graph through six MCP tools: knowledge/search, knowledge/context, knowledge/impact, knowledge/history, knowledge/remember, knowledge/recall.

Event bus and SSE bus

  • Event bus -- in-memory pub/sub for repositories and services to publish domain events.
  • SSE bus -- Server-Sent Events for live updates to the Web dashboard. sse-bus.ts is the in-memory implementation used locally; sse-redis.ts is the Redis-backed variant used in hosted mode for multi-instance deployments.

Key decisions

  • Dependency injection over globals -- AppContext is the root; every function takes it explicitly. No getApp() calls in new code.
  • DatabaseAdapter abstraction -- same repositories and stores for SQLite (local) and Postgres (hosted). No ORM.
  • Polymorphism for parsers and executors -- runtime-specific behavior lives in dedicated classes registered on AppContext.
  • ArkD as universal daemon -- one HTTP API for every compute target. No SSH, no platform-specific transport.
  • MCP socket pooling -- share MCP processes across sessions, unix sockets, 85-90% memory reduction.
  • Workdir-based transcript identification -- no "latest by mtime" races. Each runtime has a deterministic way to map a session to its transcript file.
