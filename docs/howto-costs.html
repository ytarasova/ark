How-To: Cost Tracking and Budgets

Track token usage, set budgets, manage subscription runtimes.

Overview

Ark tracks cost universally across all runtimes and providers. Every session produces usage records regardless of whether you are running Claude Code, Codex, or Gemini CLI, and regardless of whether you are paying per-token or on a flat subscription. The two moving parts are the PricingRegistry (prices for 300+ models via the LiteLLM JSON catalog) and the UsageRecorder (writes rows to the usage_records table).

Usage records carry multi-dimensional attribution: session, user, tenant, model, provider, runtime, and agent role. That means you can slice costs any way you want -- per model, per tenant, per agent role -- without having to write joins by hand.

Three cost modes

The cost_mode column on usage_records captures the billing mode that applied when the record was written:

ModeSemanticsExample
apiPer-token pricing resolved via PricingRegistry. cost_usd is the real dollar cost.Claude API, OpenAI API, Gemini API.
subscriptionFlat-rate plan. cost_usd = 0 on each record, but tokens are still counted so you can watch rate-limit exposure.Claude Max ($200/mo plan).
freeNo cost, no rate tracking.Local open-source models via a self-hosted gateway.

A single tenant can have sessions running in all three modes simultaneously. Cost reports are mode-aware and will highlight the split.

Runtime billing config

Billing mode is declared per runtime in the runtime YAML. The built-in runtimes look like this:

# runtimes/claude.yaml
name: claude
type: claude-code
billing:
  mode: api
  transcript_parser: claude

# runtimes/claude-max.yaml
name: claude-max
type: claude-code
billing:
  mode: subscription
  plan: claude-max-200
  cost_per_month: 200
  transcript_parser: claude

# runtimes/codex.yaml
name: codex
type: cli-agent
billing:
  mode: api
  transcript_parser: codex

# runtimes/gemini.yaml
name: gemini
type: cli-agent
billing:
  mode: api
  transcript_parser: gemini

Override at dispatch with --runtime. For example, to run the implementer role on Claude Max instead of the default per-token Claude runtime:

$ ark session start --repo . --summary "Fix bug" \
    --agent implementer --runtime claude-max --dispatch

How tokens are extracted

Ark uses polymorphic transcript parsers, one per runtime, registered on the TranscriptParserRegistry. Each parser knows where its runtime stores conversation logs and how to identify the session deterministically by working directory (never "latest file by mtime").

  • ClaudeTranscriptParser -- reads the exact path ~/.claude/projects/<slug>/<session>.jsonl via the stored claude_session_id.
  • CodexTranscriptParser -- scans ~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl, matches the session_meta.payload.cwd field to the session's workdir.
  • GeminiTranscriptParser -- scans ~/.gemini/tmp/<slug>/chats/session-*.jsonl, matches projectHash = sha256(workdir) against the file's first line.

Each parser walks the JSONL, extracts token counts from the assistant turns, resolves model pricing, and feeds a UsageRecorder.record() call. The record is keyed by session, turn number, and timestamp so re-parsing is idempotent.

Web dashboard

The Web dashboard has a dedicated Costs page powered by Recharts. You get a pie chart of cost by model, a bar chart of cost per day, and a filterable table by session. The Dashboard page includes a summary widget.

TUI dashboard

The TUI dashboard (ark tui) shows ASCII cost charts on the Dashboard tab and a per-session cost column on the Sessions tab.

Setting budgets

Configure budgets in ~/.ark/config.yaml:

budgets:
  dailyLimit: 50
  weeklyLimit: 200
  monthlyLimit: 500

Ark warns at 80% of any limit and blocks new dispatches at 100%. You can run a manual check any time:

$ ark costs --check-budget
today:   $12.40 / $50.00  (25%)   OK
week:    $84.10 / $200.00 (42%)   OK
month:  $412.80 / $500.00 (82%)   WARNING – approaching limit

Backfilling historical sessions

If you dispatched sessions before cost tracking existed, or ran Ark without the UsageRecorder wired, you can backfill:

$ ark costs-sync

This scans transcripts for every known runtime in their canonical locations, matches them to sessions in ark.db, and writes any missing usage_records rows. It is idempotent -- sessions that already have records are skipped.

Exporting

$ ark costs-export --format csv > costs.csv
$ ark costs-export --format json > costs.json

CSV is useful for dropping into a spreadsheet or finance pipeline; JSON preserves nested attribution fields.

Multi-tenant cost attribution

In hosted mode, every usage_records row carries the tenant_id and user_id that originated the session, plus model, provider, runtime, and agent_role. This lets you answer questions like:

  • "How much did tenant X spend this month?"
  • "Which user's sessions burned the most tokens yesterday?"
  • "What is the average cost per planner agent run vs implementer?"
  • "Which model gave us the best cost-per-successful-PR this week?"

Per-tenant caps live in the tenant policy:

max_cost_per_day_usd: 100
max_cost_per_week_usd: 500
max_cost_per_month_usd: 2000

These are enforced at dispatch time by the scheduler in hosted mode. A tenant that hits its daily cap will get a polite refusal on the next dispatch until the next window opens.

Router cost tracking

When the LLM Router is enabled, every request flowing through it carries an onUsage callback that is wired to the same UsageRecorder. That means a single dispatch produces usage records whether tokens come from transcript parsing or from the router's live feed. The router path is the preferred source for real-time tracking because transcript parsers run at session completion.

The router cost feed-back is tracked in the roadmap. Core wiring is in place; full end-to-end verification against real provider APIs is in Camp 1.

Worked example

# 1. Run a few sessions on different runtimes
$ ark session start --repo . --summary "task 1" \
    --runtime claude --dispatch
$ ark session start --repo . --summary "task 2" \
    --runtime claude-max --dispatch
$ ark session start --repo . --summary "task 3" \
    --runtime codex --dispatch

# 2. Let them finish, then backfill in case live tracking missed anything
$ ark costs-sync

# 3. View overall costs
$ ark costs

# 4. Slice by model
$ ark costs --by model

# 5. Set a daily cap so the next one of these doesn't blow the budget
$ echo 'budgets: { dailyLimit: 10 }' >> ~/.ark/config.yaml
$ ark costs --check-budget
