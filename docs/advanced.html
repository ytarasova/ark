Advanced Features

Git worktrees, knowledge graph, search, scheduling, integrations, guardrails, observability, and more.

Git Worktrees

When a session is dispatched, Ark automatically creates a git worktree at ~/.ark/worktrees/<sessionId>/. The agent works in this isolated copy of the repo, so your main checkout is never modified by a running session.

Worktree Commands

$ ark worktree list                          # list all active worktrees
$ ark worktree diff s-1a2b                    # preview changes (--base to set base branch)
$ ark worktree finish s-1a2b                  # merge or create PR
$ ark worktree pr s-1a2b                      # create PR without merging
$ ark worktree cleanup s-1a2b                 # remove worktree (--dry-run to preview)

ark worktree finish

Merges the session branch into the base branch, or creates a PR if the repo has a remote. Options:

OptionDescription
--into <branch>Target branch to merge into (default: main)
--no-mergePush the branch but skip the local merge
--keep-branchDo not delete the session branch after merging
--prForce PR creation even when a local merge is possible

ark worktree pr

Push the session branch and open a GitHub pull request.

$ ark worktree pr s-1a2b --title "Add OAuth support" --draft
OptionDescription
--title <text>PR title (default: session summary)
--base <branch>Base branch for the PR
--draftCreate a draft PR

Auto-PR

When an agent completes and the repo has a git remote, Ark automatically pushes the session branch and creates a GitHub PR via gh pr create. No manual steps required.

To disable auto-PR for a specific repo, add the following to your .ark.yaml:

auto_pr: false

TUI Workflow

In the TUI Sessions tab, press W to open the worktree finish overlay. It shows a diff preview and lets you choose between merge and PR creation without leaving the TUI.

Knowledge Graph

Ark maintains a unified knowledge graph that spans your codebase, session history, memories, and learnings. This replaces the old memories.json and LEARNINGS.md files -- all knowledge is stored as nodes and edges in the database.

How It Works

At dispatch time, the context builder queries the knowledge graph for nodes relevant to the session summary and injects them into the agent's task prompt (token-budgeted, 2000 tokens max). This includes memories, learnings from prior sessions, and relevant file/symbol information. Agents can also query the knowledge graph during execution via six MCP tools: knowledge/search, knowledge/context, knowledge/impact, knowledge/history, knowledge/remember, and knowledge/recall.

The codebase indexer is ops-codegraph (@optave/codegraph) -- a TypeScript + Rust native engine that parses 33 languages via tree-sitter WASM. No Python dependency. Install with bun add @optave/codegraph or npm install -g @optave/codegraph. The command is codegraph build; Ark reads the resulting .codegraph/graph.db and streams nodes + edges into the tenant-scoped knowledge + knowledge_edges tables. Local compute honors the knowledge.auto_index config toggle; remote compute always indexes via POST /codegraph/index on the remote arkd daemon.

Knowledge Commands

$ ark knowledge search "authentication"       # search all node types
$ ark knowledge index --repo .                 # index codebase into graph
$ ark knowledge stats                          # show node/edge counts
$ ark knowledge remember "Always use transactions" --tags db,safety
$ ark knowledge recall "database"              # search memories + learnings
$ ark knowledge export --dir ./backup
$ ark knowledge import --dir ./backup

Memory Commands (Backward-Compatible)

$ ark memory list                              # list memory nodes
$ ark memory add "Important note" --tags tag1  # add a memory
$ ark memory recall "keyword"                  # search memories
$ ark memory forget <id>                       # delete a memory

TUI and Web

In the TUI, press Y to open the Memory Manager overlay. Use j/k to navigate, n to add a new memory, and x to delete. The Web dashboard has a dedicated Memory view showing knowledge stats and memory management.

Search

Ark provides three layers of search: session metadata, Claude Code transcripts, and hybrid search with LLM re-ranking.

Session Search

$ ark search "authentication bug"
$ ark search "OAuth" --transcripts --limit 50
$ ark search "login" --index
OptionDefaultDescription
--transcriptsoffAlso search Claude Code JSONL transcripts
--indexoffRebuild the FTS5 index before searching
--limit20Maximum number of results to return

Global Conversation Search

Search across all Claude Code conversations on disk, not just Ark-managed sessions:

$ ark search-all "migration strategy"
$ ark search-all "deploy" --days 7 --limit 100
OptionDescription
--days <n>Limit to conversations from the last N days
--limit <n>Maximum number of results

Hybrid Search

Combines memory recall, knowledge base lookup, and transcript search, then re-ranks results with an LLM for relevance. Requires ANTHROPIC_API_KEY.

$ ark search "rate limiting approach" --hybrid

FTS5 Index

Build or rebuild the full-text search index over all transcripts in ~/.claude/projects/:

$ ark index
Note: The FTS5 index is stored inside ~/.ark/ark.db. Run ark index after importing many sessions or when search results appear stale. Search falls back to file scanning if the index does not exist.

Scheduled Sessions

Ark can automatically create and dispatch sessions on a cron schedule. This is useful for nightly checks, recurring reports, or any periodic task you want an agent to handle.

$ ark schedule add --cron "0 2 * * *" --summary "Nightly dependency audit" --repo . --flow bare
$ ark schedule list
$ ark schedule enable sch-1a2b
$ ark schedule disable sch-1a2b
$ ark schedule delete sch-1a2b

Cron format: minute hour day-of-month month day-of-week. Supports wildcards (*), ranges (1-5), steps (*/5), and comma lists (1,3,5).

Tip: Combine schedules with the bare flow and a focused summary to create lightweight autonomous monitors that run in the background without human gates.

Messaging Bridges

Connect Ark to external chat platforms so you can receive notifications and issue commands from Telegram, Slack, or Discord.

Configuration

Add a ~/.ark/bridge.json file with your platform credentials:

{
  "telegram": {
    "botToken": "123456:ABC-DEF...",
    "chatId": "987654321"
  },
  "slack": {
    "webhookUrl": "https://hooks.slack.com/services/..."
  },
  "discord": {
    "webhookUrl": "https://discord.com/api/webhooks/..."
  }
}

Platform Setup

PlatformHow to obtain credentials
TelegramCreate a bot via @BotFather, copy the token. Get your chat ID by messaging @userinfobot.
SlackCreate an Incoming Webhook in your Slack app settings and copy the webhook URL.
DiscordIn your Discord server settings, go to Integrations > Webhooks and create a new webhook.

Bridge Commands

$ ark conductor bridge      # start listening for incoming chat commands
$ ark conductor notify "Deployment session completed"   # send a one-off message

Profiles

Profiles provide isolated session namespaces. Use them to keep work, personal, and team sessions separate without managing multiple Ark installations.

$ ark profile list
$ ark profile create work
$ ark profile delete work

Scope commands to a profile with the -p flag:

$ ark -p work session list
$ ark -p work session start --repo . --summary "Add payment flow" --dispatch

Sessions created under a profile are scoped via a group name prefix, keeping them hidden from the default unscoped view.

Guardrails

Guardrails are pattern-based tool authorization rules that block or warn on dangerous commands before an agent can execute them. They are enforced via the PreToolUse hook in .claude/settings.local.json -- regardless of the agent's permission mode.

Default Rules

PatternActionReason
rm -rf /BlockRecursive root deletion
DROP TABLE / DROP DATABASEBlockDestructive SQL
:(){ :|:& };:BlockFork bomb
mkfs / fdisk / dd if=BlockDisk format commands
git push --forceBlockForce push to remote
.env file accessWarnPotential secrets exposure
Credentials file accessWarnPotential secrets exposure

Custom Rules

Add custom rules in your agent YAML or flow configuration. Each rule specifies a tool name, a regex pattern, and an action:

{
  "tool": "Bash",
  "pattern": "curl.*\\|.*sh",
  "action": "block"
}

Supported actions: block (abort the tool call), warn (log and continue).

Prompt Injection Guard

Ark scans task summaries and inbound messages for common prompt injection patterns before they reach the agent.

How It Works

  • At dispatch: if the summary contains a high-severity pattern, the dispatch is blocked entirely.
  • At send: if a message contains a high-severity pattern, it is blocked. Lower-severity matches are logged as warnings and allowed through.

Detection Patterns

PatternSeverity
"ignore previous instructions"High
"you are now a different"High
Fake <system> / <prompt> tagsMedium - High
"pretend you are"Medium
Important: The injection guard is a lightweight heuristic layer. It catches common patterns but is not a security boundary. Do not rely on it to safely process untrusted user input in production pipelines.

OTLP Observability

Ark can export session and stage lifecycle events as OpenTelemetry traces to any OTLP HTTP collector (Jaeger, Tempo, Honeycomb, Datadog, etc.).

Configuration

Add the following to ~/.ark/config.yaml:

otlp:
  enabled: true
  endpoint: "http://localhost:4318/v1/traces"
  headers:
    x-honeycomb-team: "your-api-key"

Trace Structure

Each session produces a root span. Each flow stage is a child span nested under the session span. Spans carry the following tags:

  • session.id, session.flow, session.repo
  • stage.name, stage.agent
  • tokens.input, tokens.output, cost.usd

Spans are buffered in memory and flushed to the collector every 30 seconds. Flush failures are silent -- trace export is fire-and-forget and will not affect session execution.

Auto-Rollback

Ark can monitor merged PRs for CI failures and automatically create revert PRs if the build breaks after merging.

Configuration

rollback:
  enabled: true
  timeout: 600          # seconds to wait for CI (default: 600)
  on_timeout: rollback   # rollback | ignore
  auto_merge: false      # auto-merge the revert PR (default: false)
  health_url: null        # optional URL to probe after CI passes

How It Works

  1. After a PR merges, Ark records the merge commit SHA.
  2. GitHub check suites are polled every 30 seconds on that SHA.
  3. If any check fails (or timeout is exceeded and on_timeout is rollback), Ark creates a revert-<branch> PR automatically.
  4. If health_url is set, Ark also probes that URL after CI passes and triggers rollback if it returns a non-2xx response.
  5. If auto_merge is true, the revert PR is merged without human approval.
Caution: Setting auto_merge: true means broken CI will result in automatic code changes to your main branch. Use only in pipelines you trust fully.

Issue Watching

Ark can watch a GitHub repository for newly opened issues with a specific label and automatically create and dispatch a session for each one.

$ ark watch --label ark --dispatch
$ ark watch --label ark --dispatch --interval 30000   # poll every 30s
OptionDefaultDescription
--label <name>--GitHub issue label to watch for
--dispatchoffAutomatically dispatch the created session
--interval <ms>60000Polling interval in milliseconds
Tip: Label your GitHub issues with ark to route them directly into your agent pipeline. Combine with a quick-fix recipe for a zero-click triage workflow.

Authentication

For local sessions, Claude Code uses your existing login. For remote sessions on EC2 or Docker compute, Ark needs a persistent OAuth token that can be synced to the remote host.

$ ark auth                   # set up credentials locally
$ ark auth --host my-ec2     # set up credentials on a remote compute host

Running ark auth invokes claude setup-token and saves the resulting OAuth token to ~/.ark/claude-oauth-token. The token is automatically synced to remote hosts during ark compute provision and ark compute sync.

Note: Run ark auth once before dispatching your first remote session. The token persists across reboots -- you do not need to re-run it unless you revoke your Claude credentials.

ArkD (Agent Daemon)

ArkD is a stateless HTTP daemon that runs on every compute target -- local, Docker, and EC2. It provides a uniform interface for agent lifecycle management, file operations, metrics collection, and channel relay regardless of the underlying provider.

What ArkD Does

  • Agent lifecycle -- launch, kill, and query agent status on the compute target
  • File operations -- read, write, and sync files on the remote host
  • Metrics -- collect CPU, memory, disk, and network metrics from the host
  • Channel relay -- forward MCP channel messages between agents and the conductor

Running ArkD

ArkD is started automatically by remote providers during provisioning. For manual use:

$ ark arkd                                          # default: port 19300, bind 0.0.0.0
$ ark arkd --port 19400 --conductor-url http://remote:19100

Architecture

The communication flow is: Agent → MCP Channel → ArkD → Conductor. On local compute, channel.ts reports directly to the conductor. On remote compute, channel.ts reports to ArkD first, and ArkD forwards to the conductor. This extra hop handles network boundaries between compute targets and the control plane.

ArkD is stateless -- it does not persist any data. All state lives in the conductor's SQLite database. Restarting ArkD has no effect on running sessions.

Fan-Out and Sub-Agent Spawning

Ark supports decomposing tasks into parallel child sessions. This enables fan-out patterns where a parent session spawns N children that work concurrently, with the parent auto-joining when all children complete.

Manual Spawning

$ ark session spawn s-1a2b "Write unit tests" --agent worker --dispatch
$ ark session spawn s-1a2b "Fix lint errors" --model haiku --dispatch
$ ark session join s-1a2b            # wait for all children
$ ark session join s-1a2b --force    # join even if children are incomplete

Automatic Fan-Out

Use a fan-out or dag-parallel flow to enable automatic task decomposition. The planner outputs a JSON array of tasks, and Ark spawns child sessions automatically. See Workflows for flow YAML syntax.

Child Session Behavior

  • Children inherit the parent's repo, compute target, and worktree
  • Each child gets its own tmux session and agent process
  • The conductor monitors child status and auto-joins when all complete
  • If any child fails, the parent is notified but not automatically failed
