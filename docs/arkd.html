ArkD - Universal Agent Daemon

HTTP daemon that runs on every compute target, bridging the control plane and the agents.

What is ArkD?

ArkD is a stateless HTTP server that runs on every compute target Ark dispatches to. It exposes a single HTTP API for agent lifecycle, file operations, process execution, metrics, and channel relay. The daemon listens on port 19300 by default and provides identical endpoints regardless of whether the target is your laptop, a Docker container, a devcontainer, an EC2 instance, a Firecracker micro-VM, or a Kubernetes pod sidecar.

Source: packages/arkd/server.ts.

Why ArkD exists

Without ArkD, the control plane would have to shell into every compute target over SSH to launch agents, read files, run commands, or report status. That approach is slow, auth-fragile, and platform-specific -- each provider would need its own transport layer.

Instead, every compute target runs arkd. The conductor and the rest of the control plane talk to arkd over HTTP. The same API works whether the compute is local, Docker, EC2, Kubernetes, or Firecracker. One transport, one protocol, every target.

Responsibilities

  • Agent lifecycle -- launch, kill, and query status of agents running in tmux sessions on the target.
  • File operations -- read, write, and list files on the target filesystem.
  • Process execution -- run commands via exec, optionally inside a sandbox.
  • System metrics -- CPU, memory, disk, uptime. Used by the dashboard and compute health checks.
  • Port probing -- check which ports are listening on the target.
  • Channel relay -- accept channel reports from agents and forward them to the conductor at http://<conductor>:19100.
  • Docker container ops -- list and exec into Docker containers on the target (used by docker-in-docker compute modes).
  • Codegraph indexing -- POST /codegraph/index runs codegraph build locally on the target, reads .codegraph/graph.db, and returns nodes and edges to the conductor.
  • Bearer token auth -- when ARK_ARKD_TOKEN is set, all requests must include a matching Authorization: Bearer <token> header.

Deployment modes

  • Local machine -- start with ark arkd, or let the server daemon auto-start it on demand.
  • Docker / devcontainer -- arkd is baked into the image and starts automatically when the container boots.
  • EC2 instance -- installed via cloud-init on first boot, started as a systemd service.
  • Kubernetes pod -- runs as a sidecar container next to the agent container.
  • Firecracker micro-VM -- baked into the micro-VM image, starts on boot.

In every case, the conductor reaches arkd at http://<target-ip>:19300. Local providers use http://localhost:19300.

Channel relay

Ark uses arkd as the HTTP bridge between agents and the conductor. The data flow:

  1. The session agent posts reports to the ark-channel MCP server, which runs inside the agent process over stdio.
  2. ark-channel forwards the report to arkd via HTTP on port 19300.
  3. ArkD relays the report to the conductor at http://<conductor>:19100.

This is why every compute target runs arkd. It gives the control plane a single HTTP endpoint per target (port 19300) regardless of how many sessions are active on it. Direct agent-to-conductor connections would not work through NAT, firewalls, or cross-cloud routing.

Human steering (from the Web UI or Desktop app) flows in the reverse direction: conductor -> arkd -> ark-channel -> agent MCP notification.

Codegraph endpoint

The /codegraph/index endpoint powers the auto-index-on-dispatch flow for remote compute. When a session is dispatched to a remote target, the control plane calls this endpoint to build a knowledge graph on the remote machine before the agent starts working.

Request body:

repoPath: string
incremental: boolean (optional)

What it does:

  1. Runs codegraph build in the repo path on the remote target.
  2. Reads the resulting .codegraph/graph.db via bun:sqlite.
  3. Returns { ok: true, nodes: [...], edges: [...], files: N, symbols: N } to the conductor.
  4. The conductor ingests the nodes and edges into the tenant-scoped knowledge and knowledge_edges tables.

Remote indexing always runs this endpoint regardless of the knowledge.auto_index config flag. The control plane needs centralized knowledge for all workers so downstream agents can query a consistent graph. Local mode honors the config flag.

Starting ArkD

# default port 19300, default conductor http://localhost:19100
$ ark arkd

# custom port
$ ark arkd --port 8080

# set conductor URL (useful when running in a container or remote host)
$ ark arkd --conductor-url http://conductor.local:19100

Environment variables:

  • ARK_ARKD_PORT -- listen port (default 19300)
  • ARK_ARKD_URL -- arkd base URL used by channel.ts and executors (default http://localhost:19300)
  • ARK_ARKD_TOKEN -- bearer token required on all incoming requests when set
  • ARK_CONDUCTOR_URL -- conductor URL arkd relays to (default http://localhost:19100)

HTTP API reference

GET/healthLiveness / readiness check
POST/execExecute a shell command
POST/agent/launchLaunch an agent via tmux
POST/agent/killKill a running agent

POST/files/readRead a file
POST/files/writeWrite a file
GET/files/listList directory contents
GET/metricsSystem metrics (CPU, memory, disk, uptime)
