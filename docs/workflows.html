Workflows

Flows, agents, skills, recipes, and the executor system.

Flows

A flow is a YAML-defined pipeline of stages. Each stage specifies an agent, a gate that controls advancement, and optional failure handling and verification scripts.

Flow Schema

name: my-flow
description: "Custom workflow"
stages:
  - name: plan
    agent: planner
    gate: manual          # auto | manual | review | condition
    on_failure: notify    # notify | retry(N)
    artifacts: [PLAN.md]   # files passed to next stage

  - name: implement
    agent: implementer
    gate: auto
    on_failure: "retry(3)"   # retry up to 3 times
    verify:
      - "npm test"
      - "npm run lint"

  - name: review
    agent: reviewer
    gate: review           # waits for GitHub PR approval

Gate Types

The gate field controls how a stage advances to the next:

GateBehavior
autoAdvances automatically when the agent completes without errors
manualWaits for human approval -- press Enter in the TUI or run ark session advance
reviewWaits for a GitHub PR approval via webhook
conditionEvaluates an expression before advancing

Verification Gates

Stages can require scripts to pass before completion. Add a verify list to any stage:

stages:
  - name: implement
    agent: implementer
    gate: auto
    verify:
      - "npm test"
      - "npm run lint"

If any script fails, the stage is blocked and the agent is steered to fix the issue. Verification can also be set in .ark.yaml as a default for all stages in a repo.

Manage verification and todos from the CLI:

$ ark session verify s-a1b2c3          # run verification manually
$ ark session todo add s-a1b2c3 "Write migration docs"
$ ark session todo list s-a1b2c3
$ ark session todo done s-a1b2c3 1      # toggle todo #1
$ ark session complete s-a1b2c3         # blocked if todos/scripts fail
$ ark session complete s-a1b2c3 --force # override verification

Todos are shown in the TUI detail panel. Press V to run verification from the TUI.

Fail-Loopback

When a stage fails, Ark can retry it automatically with the error context injected into the agent's prompt. Configure with on_failure: "retry(3)" (up to 3 retries). After exhausting retries, the session moves to failed status.

Managing Flows

$ ark flow list              # list all flows (builtin + custom)
$ ark flow show default      # show flow stages and gates

Custom flows go in flows/definitions/<name>.yaml (project) or ~/.ark/flows/<name>.yaml (global).

Parallel Execution

Ark supports three parallel execution patterns in flows: fork, fan-out, and DAG.

Fork Stages

A type: fork stage splits work into parallel child sessions based on a strategy. The parent session waits until all children complete, then automatically advances.

stages:
  - name: implement
    type: fork
    agent: implementer
    strategy: plan           # reads PLAN.md to split into subtasks
    max_parallel: 4          # max concurrent child sessions
    gate: auto

The parallel builtin flow uses this pattern: plan > fork/join implement > review > PR > merge.

Fan-Out Stages

A type: fan_out stage decomposes a task into independent parallel children. Unlike fork, fan-out stages expect the planner to produce a JSON array of tasks. Each task becomes a child session that runs independently.

stages:
  - name: plan
    agent: planner
    gate: manual
    task: |
      Decompose into sub-tasks. Output JSON: [{"summary": "...", "agent": "implementer"}, ...]
  - name: execute
    type: fan_out
    gate: auto
  - name: review
    agent: reviewer
    gate: auto

DAG Dependencies

Stages can declare dependencies with depends_on, forming a directed acyclic graph (DAG). Independent stages run in parallel; dependent stages wait. This enables patterns like parallel implement + test, then integrate.

stages:
  - name: plan
    agent: planner
    gate: manual

  - name: implement
    agent: implementer
    gate: auto
    depends_on: [plan]

  - name: test
    agent: implementer
    gate: auto
    depends_on: [plan]        # runs in parallel with implement

  - name: integrate
    agent: implementer
    gate: auto
    depends_on: [implement, test]  # waits for both

  - name: review
    agent: reviewer
    depends_on: [integrate]

The dag-parallel builtin flow uses this pattern.

Auto-Join

When the last child session in a fork group completes, Ark automatically joins the results back into the parent. The parent transitions from waiting back to ready and advances to the next stage. If any children fail, a fan_out_partial_failure event is logged but the parent still advances.

CLI Commands for Parallel Work

$ ark session spawn s-parent "Write unit tests" --dispatch
$ ark session spawn s-parent "Fix linting" --agent worker --dispatch
$ ark session join s-parent        # manual join (auto-join handles this)
$ ark session fork s-parent        # fork conversation for a divergent approach

Agents

Agents are YAML-defined Claude Code configurations. They specify the model, system prompt, tools, and permission mode. At dispatch, template variables in the system prompt are substituted with live session values.

Three-Tier Resolution

Ark resolves agent definitions in priority order (highest first):

  1. Project: .ark/agents/<name>.yaml in the repo
  2. Global: ~/.ark/agents/<name>.yaml
  3. Builtin: agents/<name>.yaml shipped with Ark

Project agents override builtins of the same name, and global agents override both.

Agent Schema

name: my-agent
description: "What this agent does"
model: opus              # opus | sonnet | haiku
max_turns: 200
system_prompt: |
  Working on {repo}. Task: {summary}. Ticket: {ticket}.
  Branch: {branch}. Workdir: {workdir}.
tools:
  - Bash
  - Read
  - Write
  - Edit
  - Glob
  - Grep
  - WebSearch
permission_mode: bypassPermissions
mcp_servers: []          # MCP server names to attach
skills: [code-review]    # skills to inject at dispatch
memories: []             # memory file paths
env:
  CLAUDE_AUTOCOMPACT_PCT_OVERRIDE: "80"

Template variables substituted at dispatch: {ticket}, {summary}, {workdir}, {repo}, {branch}.

Builtin Agents

AgentModelPurpose
plannerSonnetCreates PLAN.md with architecture and implementation strategy
implementerOpusWrites code, tests, and commits
reviewerSonnetReviews PRs and produces structured JSON feedback (P0-P3)
documenterSonnetGenerates project documentation
workerOpusGeneral-purpose lightweight agent

Managing Agents

$ ark agent list                    # list all agents (builtin + custom)
$ ark agent show implementer        # show agent details
$ ark agent create my-agent         # create new agent (opens editor)
$ ark agent edit my-agent           # edit agent definition
$ ark agent copy implementer fast   # copy agent for customization
$ ark agent delete my-agent         # delete a custom agent

Executor System

Agents dispatch through pluggable executors. The runtime field in agent YAML selects which executor launches the agent. This lets you orchestrate Claude Code, custom subprocess tools, or any CLI coding tool through the same session and lifecycle system.

Built-in Executors

RuntimeDescription
claude-codeDefault. Launches Claude Code in a tmux session with hooks and an MCP channel for bidirectional messaging.
subprocessSpawns any command as a child process. Suitable for scripts, linters, and build tools.
cli-agentRuns any CLI coding tool (codex, gemini, aider, etc.) in tmux with worktree isolation, the same way Claude Code agents run.

Subprocess Agent

Use the subprocess runtime to wrap any script or tool as an agent:

name: my-linter
description: "Run project lint checks"
runtime: subprocess
command: ["node", "scripts/lint.js"]
env:
  TARGET: "{workdir}"

CLI Agents

The cli-agent runtime lets Ark orchestrate any CLI coding tool. CLI agents get the same worktree isolation, session tracking, and lifecycle management as Claude Code agents -- the only difference is which tool is launched.

Builtin CLI Agents

AgentToolDescription
codex-workerOpenAI Codex CLIFull-auto Codex coding agent
gemini-workerGoogle Gemini CLIGemini coding agent
aider-workerAiderAider chat-based coding
generic-cliAny CLIGeneric wrapper for custom tools

CLI Agent Schema

name: codex-worker
description: OpenAI Codex CLI coding agent
runtime: cli-agent
command: ["codex", "--approval-mode", "full-auto"]
task_delivery: arg       # stdin | file | arg (default)
model: o4-mini
max_turns: 200
system_prompt: ""
tools: []
permission_mode: bypassPermissions
env: {}

Task Delivery Modes

The task_delivery field controls how the task is passed to the CLI tool:

ModeBehaviorExample
stdinPipes the task to stdinecho "task" | codex
fileWrites task to a temp file and passes the pathcodex --task /tmp/task.txt
argAppends the task as the last CLI argument (default)codex "Fix the login bug"

Usage

$ ark session start --repo . --summary "Fix the bug" --agent codex-worker --dispatch
$ ark agent create my-tool   # set runtime: cli-agent in the YAML

Skills

Skills are reusable prompt fragments injected into agent system prompts at dispatch time. They encapsulate domain knowledge, coding patterns, and procedural instructions that agents carry across sessions.

Three-Tier Resolution

Skills are resolved in priority order (highest first):

LocationScope
.ark/skills/<name>.mdProject-local (committed to repo)
~/.ark/skills/<name>.mdGlobal (user-wide)
skills/<name>.mdBuiltin (shipped with Ark)

Attaching Skills to Agents

Add skill names to the skills field in agent YAML. At dispatch, Ark automatically injects the content of each listed skill into the agent's system prompt:

name: my-agent
skills:
  - code-review
  - test-writing

Managing Skills

$ ark skill list                   # list all skills (builtin + custom)
$ ark skill show code-review       # show skill content

$ ark skill create my-skill -d "Review checklist" -p "Always check for..."
$ ark skill create --from my-skill.yaml
$ ark skill create my-skill -d "..." -p "..." --scope project --tags "review,quality"

$ ark skill delete my-skill
$ ark skill delete my-skill --scope project

Project-scoped skills are saved to .ark/skills/ and committed with the repo, making them available to all sessions started from that project.

Skill Extraction

When a session completes, Ark automatically analyzes the conversation transcript for reusable patterns -- multi-step procedures, repeated methodology, and structured approaches. High-confidence candidates are saved as global skills with names like extracted-<sessionId>-0. Extraction runs silently and never blocks session completion.

Recipes

Recipes are session templates with variables. They let you quick-launch sessions from presets without manually configuring flow, agent, summary, and compute each time.

Three-Tier Resolution

Recipes use the same three-tier resolution as skills (highest priority first):

LocationScope
.ark/recipes/<name>.yamlProject-local
~/.ark/recipes/<name>.yamlGlobal
recipes/<name>.yamlBuiltin

Recipe Schema

name: quick-fix
description: "Fast bugfix with bare flow"
repo: .
flow: bare
summary: "Fix: {issue}"
variables:
  - name: issue
    prompt: "Describe the issue"

Using Recipes

$ ark recipe list
$ ark recipe show quick-fix
$ ark session start --recipe quick-fix --repo . --dispatch

Builtin Recipes

RecipeFlowPurpose
quick-fixbareFast single-agent bugfix
feature-buildquickImplement and open a PR
code-reviewpr-reviewReview an existing PR
fix-bugdefaultFull pipeline bugfix
new-featuredefaultFull SDLC for a new feature

Creating Recipes

$ ark recipe create --from my-recipe.yaml
$ ark recipe create --from-session s-a1b2c3 --name my-recipe
$ ark recipe create --from my-recipe.yaml --scope project

$ ark recipe delete my-recipe
$ ark recipe delete my-recipe --scope project

You cannot delete builtin recipes. Use --scope project or --scope global to target a specific tier when names conflict.
