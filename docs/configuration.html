Local vs Hosted Mode

Ark runs in one of two deployment modes. Most config keys apply to both, but a few are mode-specific.

~/.ark/config.yaml

The main configuration file. Open it with ark config (creates a default if missing).

Budget Limits

Set spending limits to prevent runaway costs. All values in USD.

budgets:
  dailyLimit: 50       # USD per day
  weeklyLimit: 200     # USD per week
  monthlyLimit: 500    # USD per month

Omit a field to have no limit for that period. When a limit is approached or exceeded, Ark displays a warning before dispatching.

OTLP (OpenTelemetry Traces)

Export session and stage spans as OpenTelemetry traces to any OTLP/HTTP collector.

otlp:
  enabled: true
  endpoint: "http://localhost:4318/v1/traces"
  headers:
    Authorization: "Bearer ..."
FieldTypeDefaultDescription
enabledbooleanfalseEnable OTLP trace export
endpointstring--OTLP/HTTP endpoint (Jaeger, Tempo, Honeycomb, etc.)
headersobject--Optional headers (auth tokens, API keys)

One trace per session with child spans per flow stage. Spans include session ID, agent, stage name, duration, and cost.

Auto-Rollback

Automatically revert merged PRs when CI fails after merge.

rollback:
  enabled: false
  timeout: 600          # seconds to wait for CI
  on_timeout: ignore    # rollback | ignore
  auto_merge: false      # auto-merge the revert PR
  health_url: null       # optional URL to probe after CI passes
FieldTypeDefaultDescription
enabledbooleanfalseActivate post-merge monitoring
timeoutnumber600Seconds to wait for CI check suites
on_timeoutstring"ignore""rollback" creates a revert PR, "ignore" does nothing
auto_mergebooleanfalseAuto-merge the revert PR
health_urlstringnullURL to poll after CI passes (non-2xx triggers rollback)

Default Compute

default_compute: my-ec2

Name of a provisioned compute resource to use by default. Also settable via ARK_DEFAULT_COMPUTE env var. CLI flags and per-repo .ark.yaml override this.

Notifications

notifications: true

Enable or disable desktop notifications for session events.

Telemetry

telemetry:
  enabled: false
  endpoint: null

Optional anonymous usage telemetry. Disabled by default. Can also be enabled via ARK_TELEMETRY=1.

Router

router:
  enabled: false
  url: http://localhost:8430
  port: 8430
  policy: balanced     # quality | balanced | cost
  auto_start: false   # start router automatically at boot

Configure the LLM router. Providers are auto-detected from env vars (ANTHROPIC_API_KEY, OPENAI_API_KEY, GOOGLE_API_KEY). When enabled, the router URL is injected into agent executors via ANTHROPIC_BASE_URL (Claude Code runtimes) or OPENAI_BASE_URL (Codex / OpenAI-compatible runtimes) -- agents call the router instead of the provider directly, and the router records every turn through its onUsage callback into the usage_records table.

TensorZero Gateway

tensorZero:
  enabled: false
  port: 3000
  config_dir: ~/.ark/tensorzero
  auto_start: false

Optional Rust-based OpenAI-compatible gateway (Apache 2.0) that the router delegates to. When enabled + router.auto_start are both on, the lifecycle manager tries three strategies in order: detect an existing sidecar, launch the native tensorzero binary, or fall back to Docker. The tensorzero.toml config is generated on the fly from the provider API keys in your environment.

Knowledge Graph

knowledge:
  auto_index: true       # index repo automatically on dispatch
  incremental_index: true

Local compute honors the auto_index flag. Remote compute (Docker, EC2, K8s, Firecracker, E2B) always indexes via POST /codegraph/index on the remote arkd daemon regardless of config. The indexer is ops-codegraph (@optave/codegraph) -- a TypeScript + Rust native tool that parses 33 languages via tree-sitter WASM, no Python dependency. Install with bun add @optave/codegraph or npm install -g @optave/codegraph. The command is codegraph build.

MCP Socket Pooling

mcp_pool:
  enabled: false
  autoStart: true
  poolAll: false
  excludeMcps: []

Share one MCP server process across all sessions instead of spawning N sessions × M MCP servers. Each pooled server exposes a socket at /tmp/ark-mcp-<name>.sock; sessions reference it via {"command": "ark", "args": ["mcp-proxy", "/tmp/ark-mcp-<name>.sock"]}. Cuts MCP memory roughly 85-90%. Use excludeMcps to keep stateful servers per-session.

Compute Templates

compute_templates:
  gpu-large:
    provider: ec2
    config:
      size: xxl
      region: us-west-2
  sandbox:
    provider: docker
    config:
      image: node:20
  isolated:
    provider: ec2-firecracker
    config:
      size: l

Named presets for ark compute create --from-template <name>. Provider values: local, docker, devcontainer, firecracker, ec2, ec2-docker, ec2-devcontainer, ec2-firecracker, e2b, k8s, k8s-kata. In hosted mode, templates live in the tenant-scoped compute_templates DB table and are managed via ark compute template create/list/show/delete.

Auth (Multi-Tenant)

auth:
  enabled: false
  apiKeyEnabled: false

Required for hosted mode, optional for local use. API key format: ark_<tenantId>_<secret>. The tenant ID is encoded in the key so routing is lookup-free; only the <secret> is hashed. Roles: admin (full tenant control), member (create and operate sessions/compute), viewer (read-only). Manage via ark auth create-key, ark auth list-keys, ark auth revoke-key, ark auth rotate-key.

When auth is off, everything behaves as a single-user install under tenant "default" with admin role. When auth is on, every tenant-owned entity is filtered by its tenant_id column: sessions, compute, events, messages, todos, groups, schedules, compute_pools, compute_templates, usage_records, resource_definitions, knowledge, knowledge_edges.

Tenant Policies

Hosted mode enforces per-tenant policies via TenantPolicyManager. Fields include allowed_providers, default_provider, max_sessions, max_cost_per_day, plus integration enforcement flags:

  • router_required -- force all traffic through the LLM router (block direct provider env vars)
  • auto_index_required -- force knowledge.auto_index on regardless of user config
  • router_policy -- lock the router policy to quality, balanced, or cost
  • tensorzero_enabled -- force TensorZero backend on for this tenant
ark tenant policy set acme \
  --providers k8s,e2b,ec2 \
  --default-provider k8s \
  --max-sessions 20 \
  --max-cost 100

Database

database_url: null    # PostgreSQL for hosted mode
redis_url: null       # Redis for SSE bus

Set DATABASE_URL for PostgreSQL (hosted mode). Set REDIS_URL for multi-instance SSE. Both default to local (SQLite, in-memory bus).

Channels

Each Ark session spawns an ark-channel MCP server over stdio. The channel opens an HTTP port for inbound traffic at 19200 + (hash(sessionId) % 10000) -- deterministic per session, so restarts reuse the same port. Reports flow agent -> ark-channel (stdio) -> arkd (:19300) -> conductor (:19100). Port 19100 is hardcoded; 19300 (arkd) is configurable via ARK_ARKD_PORT / ARK_ARKD_URL.

.ark.yaml (Per-Repository)

Place in your repo root. Ark reads it when starting sessions in that repo. Recognized file names (in priority order): .ark.yaml, .ark.yml, ark.yaml.

flow: quick               # default flow
group: my-team             # default group name
compute: my-ec2            # default compute resource
agent: implementer         # default agent
auto_pr: true              # auto-create PR on agent completion
env:
  CUSTOM_VAR: "value"
verify:                     # verification scripts for all stages
  - "npm test"
  - "npm run lint"
FieldTypeDescription
flowstringDefault flow name
groupstringDefault group name
computestringDefault compute resource
agentstringDefault agent name
envobjectEnvironment variables for agents
verifystring[]Verification scripts run before stage completion
auto_prbooleanAuto-create PR on agent completion (default: true for repos with remotes)

CLI flags always override .ark.yaml defaults:

$ ark session start --repo .                     # uses flow from .ark.yaml
$ ark session start --repo . --flow bare          # overrides flow to "bare"

Agent YAML

Agent definitions live in agents/<name>.yaml (builtin), .ark/agents/<name>.yaml (project), or ~/.ark/agents/<name>.yaml (global). See Agents Reference for detailed per-agent documentation.

name: my-agent
description: "What this agent does"
model: opus               # opus | sonnet | haiku
max_turns: 200
system_prompt: |
  Working on {repo}. Task: {summary}. Ticket: {ticket}.
tools: [Bash, Read, Write, Edit, Glob, Grep, WebSearch]
permission_mode: bypassPermissions
skills: [code-review]     # skills injected into system prompt
memories: []              # memory categories recalled at dispatch
context: [CLAUDE.md]      # files included as context
env:
  CLAUDE_AUTOCOMPACT_PCT_OVERRIDE: "80"

Template variables substituted at dispatch: {ticket}, {summary}, {workdir}, {repo}, {branch}.

Built-in agents (12 roles): closer, documenter, implementer, plan-auditor, planner, retro, reviewer, spec-planner, task-implementer, ticket-intake, verifier, worker. Built-in runtimes: claude, claude-max, codex, gemini. Any role can be mapped to any runtime at dispatch with --runtime.

Runtime YAML

Runtime definitions live in runtimes/<name>.yaml (builtin), .ark/runtimes/<name>.yaml (project), or ~/.ark/runtimes/<name>.yaml (global). Built-in runtimes: claude, claude-max, codex, gemini.

name: claude-max
description: "Claude Code on Max subscription"
type: claude-code
models:
  - id: opus
    label: "Claude Opus"
default_model: opus
billing:
  mode: subscription          # api | subscription | free
  plan: claude-max
  cost_per_month: 200
  transcript_parser: claude

The billing section tells the UsageRecorder how to account for token usage. Each row in usage_records carries a cost_mode column mirroring the runtime's mode:

Modecost_usd per rowTokens recordedExample runtimes
apiPer-token from PricingRegistry (300+ models, LiteLLM JSON)Yesclaude, codex, gemini
subscription0 (monthly cost amortized separately)Yes (for rate-limit tracking)claude-max ($200/mo)
free0YesLocal or zero-cost runtimes

The transcript_parser field is used by the polymorphic TranscriptParserRegistry and ark costs-sync to pick the right backend: ClaudeTranscriptParser reads ~/.claude/projects/<slug>/<sessionId>.jsonl, CodexTranscriptParser reads ~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl (cwd-matched), GeminiTranscriptParser reads ~/.gemini/tmp/<projectHash>/chats/session-*.jsonl.

CLI: ark runtime list, ark runtime show <name>. Override an agent's default runtime at dispatch with --runtime:

$ ark session start --repo . --summary "Fix bug" --agent implementer --runtime codex --dispatch

Flow YAML

Flows live in flows/definitions/<name>.yaml (builtin), .ark/flows/<name>.yaml (project), or ~/.ark/flows/<name>.yaml (global). See Flows Reference for detailed per-flow documentation.

name: my-flow
description: "Custom workflow"
stages:
  - name: plan
    agent: planner
    gate: manual           # auto | manual | review | condition
    on_failure: notify
  - name: implement
    agent: implementer
    gate: auto
    on_failure: "retry(3)"
    verify:
      - "npm test"

Built-in flows: default, quick, bare, parallel, fan-out, pr-review, dag-parallel.

Skill Definitions

Skills are reusable prompt fragments for agents. Three-tier resolution (highest priority first):

LocationScope
.ark/skills/<name>.mdProject-local (committed to repo)
~/.ark/skills/<name>.mdGlobal (user-wide)
skills/<name>.mdBuiltin (shipped with Ark)

Attach skills to agents via the skills field in agent YAML. At dispatch, skill content is automatically injected into the agent's system prompt.

Recipe Definitions

Recipes are session templates with variables. Same three-tier resolution as skills.

name: quick-fix
description: "Fast bugfix with bare flow"
repo: .
flow: bare
summary: "Fix: {issue}"
variables:
  - name: issue
    prompt: "Describe the issue"

Built-in recipes: quick-fix, feature-build, code-review, fix-bug, new-feature.

Profiles

Profiles provide isolated session namespaces. Managed by ark profile commands -- stored in ~/.ark/profiles.json.

$ ark profile list
$ ark profile create work "Work projects"
$ ark -p work session list               # scope to a profile

Cron Schedules

$ ark schedule add --cron "0 2 * * *" --flow bare --repo /path --summary "Nightly tests"

Cron format: minute hour day-of-month month day-of-week. Supports wildcards (*), ranges (1-5), steps (*/5), and comma lists (1,3,5).
