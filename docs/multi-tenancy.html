Multi-Tenancy and Multi-User

Running Ark as a hosted control plane with multiple tenants and users.

Overview

Ark ships with a single codebase that runs in two modes:

  • Local single-user mode -- the default. SQLite, file-backed stores, no auth, everything in ~/.ark/.
  • Hosted / control plane mode -- PostgreSQL, DB-backed stores, API-key auth, worker registry, session scheduler, tenant policies.

The mode is selected by configuration: set DATABASE_URL to a Postgres connection string and Ark boots in hosted mode. Everything else -- repositories, services, stores, executors -- is shared between modes.

Local mode (default)

  • Single user (you).
  • Single tenant (default).
  • SQLite at ~/.ark/ark.db in WAL mode with a 5-second busy timeout.
  • File-backed stores under ~/.ark/agents/, ~/.ark/flows/, ~/.ark/skills/, ~/.ark/recipes/, ~/.ark/runtimes/.
  • Three-tier resolution for resources: builtin > global (~/.ark/) > project (.ark/ in repo).
  • No authentication.
  • Cost tracking is local only (session-level rows in usage_records).
  • Conductor runs under the server daemon; the CLI skips the conductor and writes directly to the DB.

Start locally:

$ ark server daemon start --detach
$ ark web
$ ark session start --repo . --summary "Fix bug" --agent implementer --dispatch

Hosted / control plane mode

  • Many tenants, many users per tenant.
  • PostgreSQL via DATABASE_URL.
  • Redis via REDIS_URL for the multi-instance SSE bus.
  • API-key authentication is required (auth.enabled: true).
  • DB-backed stores: DbResourceStore reads agents, flows, skills, recipes, and runtimes from the tenant-scoped resource_definitions table.
  • Worker registry and session scheduler (hosted.ts) distribute sessions across a pool of workers.
  • Tenant policies enforce allowed providers, max concurrent sessions, daily cost caps, and integration settings (router, auto-index, TensorZero).
  • Per-tenant AppContext via app.forTenant(tenantId) gives scoped repository and store views.

Start hosted mode:

$ ark server start --hosted
# or via docker-compose
$ docker-compose -f .infra/docker-compose.yaml up -d

Tenant scoping

Every entity table in Ark has a tenant_id column. Sessions also have a user_id. Tenant-scoped tables:

  • sessions
  • compute, compute_templates, compute_pools
  • events, messages, todos
  • groups, schedules
  • usage_records
  • resource_definitions
  • knowledge, knowledge_edges

Creating a tenant-scoped view:

const scoped = app.forTenant(tenantId);
// All queries on scoped.sessions, scoped.computes, etc.
// are automatically filtered by tenant_id.
const rows = scoped.sessions.list();

RPC handlers extract tenant context from the API key and operate on a scoped AppContext. Cross-tenant access is impossible at the repository layer unless you explicitly go through the unscoped root context.

Authentication

API keys use the format ark_<tenantId>_<secret>. The ApiKeyManager stores SHA-256 hashes of the secret in the api_keys table.

Roles:

  • admin -- manage tenant policies, API keys, and all sessions within the tenant.
  • member -- create and manage their own sessions.
  • viewer -- read-only access.

CLI commands:

$ ark auth setup
$ ark auth create-key <name> --role member
$ ark auth list-keys
$ ark auth revoke <id>
$ ark auth rotate <id>

ApiKeyManager.validate(key) returns a TenantContext { tenantId, userId, role }. Remote clients pass their key via the ARK_TOKEN environment variable or the --token flag.

Tenant policies

Policies are stored per-tenant in the tenant_policies table and managed by TenantPolicyManager.

Compute policy:

  • allowed_providers -- whitelist of compute providers this tenant may use.
  • default_provider -- provider used when the user doesn't specify one.
  • max_concurrent_sessions -- hard cap on active sessions per tenant.
  • max_cost_per_day_usd -- daily cost cap; sessions are rejected when exceeded.
  • compute_pools -- named pools of preconfigured compute resources.

Integration policy:

  • router_enabled / router_required -- whether the LLM Router is used and whether the tenant can opt out.
  • router_policy -- quality / balanced / cost.
  • auto_index / auto_index_required -- whether knowledge indexing runs at dispatch and whether the tenant can opt out.
  • tensorzero_enabled -- whether LLM calls route through TensorZero.

getEffectiveIntegrationSettings(tenantId, globalConfig) merges the tenant policy with the global config fallback. The _required flags override global opt-outs so a tenant can enforce settings even when the global default is off.

Worker registry and scheduler

  • WorkerRegistry (worker-registry.ts) -- workers register with the control plane via HTTP, are health-checked every 60 seconds, and pruned after 90 seconds of staleness.
  • SessionScheduler (scheduler.ts) -- assigns new sessions to available workers, respects tenant policies (provider whitelist, max concurrent, cost cap), and routes work to compute pools when configured.
  • SSE bus -- relays session and event updates across control-plane instances via Redis so dashboards show live updates regardless of which node holds the websocket.

DB-backed resource stores

In hosted mode, agents, flows, skills, recipes, and runtimes live in the tenant-scoped resource_definitions table. DbResourceStore<T> implements the same interface as the file-backed stores (list, get, save, delete), but reads and writes rows instead of files.

Local mode continues to use file-backed three-tier resolution (builtin > global > project). You can export a local resource set to a tenant and back for portability.

Multi-user session tracking

Sessions carry a user_id populated from the API key's owner. Dashboards can filter and aggregate by user:

$ ark session list --user alice
$ ark cost summary --group-by user_id

Cost breakdowns roll up per user per tenant, so you can see which team member spent what over any time window.

Deployment

Docker Compose (.infra/docker-compose.yaml) brings up the full stack: Ark server, workers, PostgreSQL, Redis, and TensorZero.

$ docker-compose -f .infra/docker-compose.yaml up -d

Kubernetes (.infra/helm/ark/) ships a Helm chart with optional Kata and Firecracker support for secure compute isolation:

$ helm install ark .infra/helm/ark -f .infra/helm/ark/values-production.yaml
