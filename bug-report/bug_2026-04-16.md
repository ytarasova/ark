# Project Bug Report -- 2026-04-16

## Bug-013: CodeBurn dashboard ignores remote compute sessions (transcripts live on remote node) [OPEN]

**User impact:** Every user running agents on remote compute (docker, devcontainer, firecracker, ec2, ec2-docker, ec2-devcontainer, ec2-firecracker, e2b, k8s, k8s-kata) gets zero rows in `burn_turns` for those sessions. The CodeBurn tab shows empty or zero-filled totals and omits the sessions entirely from all panels -- overview, daily activity, per-project, per-model, per-category, tools, bash, and MCP servers. Failure is silent: no error is surfaced, the dashboard simply lacks data as if those sessions never ran. Anyone who does not exclusively use local compute loses all burn visibility for their remote sessions. The `BurnNote` banner in the UI (commit `bdc2b8a`) does acknowledge this with a "Local sessions only" disclaimer, but the data loss itself is unrecoverable without a code fix.
**Tracking:** https://github.com/ytarasova/ark/pull/153 (feature/burn-dashboard, introduces the local-only dashboard; banner added in commit `bdc2b8a` references this constraint)
**Verified:** 2026-04-16 via code exploration; `packages/web/src/components/burn/BurnView.tsx:17-39` contains the "Local sessions only" banner confirming this is a known architectural gap. Root cause traced through `packages/core/services/session-hooks.ts`, `packages/core/observability/burn/record.ts`, and `packages/core/observability/burn/sync.ts`.

### Expected behavior

Running an agent on any compute provider should produce burn turns in `burn_turns` keyed by session_id and turn_index. Those turns should appear in every CodeBurn panel (overview, daily activity, per-project, per-model, per-category, tools, bash, MCP servers). The compute provider used should be transparent to burn data collection.

### Steps to reproduce

1. Start the ark daemon on branch `feature/burn-dashboard` (or any branch with burn support): `ark server daemon start --detach`
2. Dispatch an agent on remote compute: `ark session start --flow quick --compute docker --repo /path/to/repo --summary "test burn collection" --dispatch`
3. Wait for the session to reach `completed` status: `ark session list`
4. Open the web dashboard at `http://localhost:5173/#/burn`, any period tab (day/week/month/all)
5. Observed: the remote session's cost, tokens, and tool usage are absent from every panel. No error -- the data silently fails to appear.
6. Run `ark costs-sync` on the conductor machine and reload -- still no data.

### Analysis of root cause

The burn pipeline has two entry paths, both of which assume the transcript file is readable from the conductor's local filesystem:

**Completion hook path (Claude runtime):** `packages/core/services/session-hooks.ts:243` calls `recordBurnTurns(app, session.id, transcriptPath, "claude", burnProject)`. The `transcriptPath` is taken directly from the hook payload at `session-hooks.ts:227` and is the agent-side absolute path. `recordBurnTurns` in `packages/core/observability/burn/record.ts:30` calls `burnParser.parseTranscript(transcriptPath, project)`. Each per-runtime parser (e.g. `packages/core/observability/burn/parsers/claude.ts`) calls `fs.readFileSync(transcriptPath)`. When the conductor and agent run on different machines, this path does not exist on the conductor -- `readFileSync` throws, the per-session try/catch in `record.ts` swallows the error, and zero turns are recorded.

**Completion hook path (Codex / Gemini runtimes):** `packages/core/services/session-orchestration.ts:1057-1059` calls `recordBurnTurns` for non-Claude runtimes after session completion. Same `readFileSync` failure mode as above.

**Manual backfill path (`syncBurn`):** `packages/core/observability/burn/sync.ts:36` iterates sessions and resolves a transcript via each runtime parser's `findForSession` method, which greps `~/.claude/projects/`, `~/.codex/sessions/`, and `~/.gemini/tmp/` respectively. Then `statSync` is called on the resolved path at `sync.ts:77-81`. All three searches look at the CONDUCTOR's home directory. Remote transcripts never exist there -- the session is silently skipped.

Arkd on the remote machine exposes `/file/read` and `/file/list` endpoints (`packages/arkd/server.ts:136-184`), and the conductor knows each remote's arkd URL via `provider.getArkdUrl` (`packages/core/conductor/conductor.ts:496-500`). However, nothing in the burn pipeline routes through arkd for remote sessions. The conductor also receives session-level totals via `usage_records` at `SessionEnd`, but those lack per-turn granularity -- no tool breakdown, no bash calls, no MCP server detail, no category, no retry counts, no speed metrics.

### Workaround

Pin burn-sensitive work to local compute. Users who need CodeBurn data should avoid docker, ec2, k8s, e2b, firecracker, and devcontainer compute targets until a fix lands. Running `ark costs-sync` on the conductor machine does NOT help -- the backfill path has the same filesystem limitation.

Longer-term options (not a fix yet):

- Option A: Have the conductor pull the transcript from remote arkd via `ArkdClient.readFile` when the session's compute is non-local, buffer it in a temp file, then run the existing parser on the buffered content. Smallest blast radius; reuses 100% of existing parser and classifier logic.
- Option B: Run the parser on the remote (arkd-side) and push classified turn rows back to a new conductor endpoint. More accurate for large transcripts but requires new arkd API surface.
- Option C: Aggregate from `usage_records` and `events` tables and give up per-turn fidelity. Least accurate but no new remote I/O needed.

See PR #153's exploration notes for option tradeoffs.
