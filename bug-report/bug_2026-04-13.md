# Project Bug Report -- 2026-04-13

## Bug-001: SessionService.start() leaves session in "pending" status, dispatch always fails [FIXED]

**Verified:** 2026-04-13 on ark v0.13.0 / Claude Code v2.1.104

### Expected behavior
`ark session start --repo . --summary "task" --flow bare --dispatch` should create a session, set its first flow stage, transition to `ready` status, and dispatch the agent.

### Steps to reproduce
1. `make install` (clean install)
2. `ark session start --repo . --summary "Add missing awilix dependency" --flow bare --dispatch`
3. Output: `Dispatch failed: Not ready (status: pending). Stop it first, or wait for it to finish.`
4. `ark session show s-6148b9` confirms `Status: pending`, `Stage: -`

### Analysis of root cause
`SessionService.start()` (`packages/core/services/session.ts:50-69`) calls `sessions.create(opts)` which defaults to `status: "pending"` and does NOT resolve the first flow stage. The comment on line 47 explicitly says "no flow-stage resolution".

Meanwhile, `dispatch()` (`packages/core/services/session-orchestration.ts:231`) requires `session.status === "ready"`.

The old `createSession()` in `session-orchestration.ts:190-204` DID resolve the first stage and set `status: "ready"`, but the new `SessionService.start()` was a simplified port that dropped this logic.

### Workaround
Added flow stage resolution to `SessionService.start()` -- after creating the session, it looks up the flow definition via `app.flows.get(flowName)`, sets `stage` to the first stage name, and updates `status` to `"ready"`. Applied in `packages/core/services/session.ts`.

## Bug-002: Claude Code launcher passes flags unsupported by Claude Code v2.1.79 (--dangerously-load-development-channels, --remote-control) [PARTIAL]

**Verified:** 2026-04-13 on ark v0.13.0 / Claude Code v2.1.104 -- crash is gone (flag is now a no-op with message "Channels are not currently available"), but `--dangerously-load-development-channels` still does not activate Channels on this account.

### Expected behavior
`ark session dispatch <id>` should launch Claude Code in a tmux session and the agent should start accepting input.

### Steps to reproduce
1. Install Claude Code v2.1.79 or earlier
2. Fix Bug-001 so dispatch proceeds
3. `ark session dispatch s-4e98f0`
4. `tmux capture-pane -t ark-s-4e98f0 -p` shows:
   ```
   error: unknown option '--dangerously-load-development-channels'
   ```
5. Agent fails to start, tmux session drops to bash prompt

### Analysis of root cause
`buildLauncher()` in `packages/core/claude/claude.ts:480-481` hardcodes two real Claude Code flags:
- `--dangerously-load-development-channels server:ark-channel` -- bypasses the channels allowlist to load a custom dev channel. Part of Claude Code Channels (March 2026 research preview). See https://code.claude.com/docs/en/channels-reference
- `--remote-control <sessionName>` -- enables Remote Control for the session. Requires Claude Code v2.1.51+. See https://code.claude.com/docs/en/remote-control

Both flags are REAL features in recent Claude Code. The problem was a version mismatch: Claude Code v2.1.79 (the version we had installed at first reproduction) did NOT support these flags yet, even though the documentation says v2.1.51+ for `--remote-control`.

### Workaround
**Upgrade Claude Code to v2.1.104 or later.** `claude --version` should show v2.1.104+. Verified working: both flags are accepted in v2.1.104.

Before the upgrade, the workaround was to remove both flags from `buildLauncher()`, but this disables Ark's channel protocol and remote control integration.

### Status
FIXED by upgrading Claude Code from v2.1.79 → v2.1.104 (released early April 2026). Not a code bug in Ark. Users on older Claude Code need to update.

## Bug-004: SkillStore does not resolve project-level .ark/skills/ directory -- project skills not visible in ark skill list [OPEN / REPRODUCED]

**Verified:** 2026-04-13 on ark v0.13.0 / Claude Code v2.1.104 -- REPRODUCED. Created `/tmp/repro-repo/.ark/skills/test-skill.yaml` with valid YAML (name, description, prompt, tags). Ran `ark skill list` from within `/tmp/repro-repo` -- output showed only 7 builtin skills, no project entries. `ark skill show test-skill` returned "Skill not found: test-skill". Bug still present in upstream.

### Expected behavior
Skills placed in `.ark/skills/<name>.yaml` within a project repo should appear in `ark skill list` with source `project`, matching the three-tier resolution documented in CLAUDE.md (builtin > global > project).

### Steps to reproduce
1. Create `.ark/skills/project-bug-report.yaml` in the project root with valid YAML (name, description, prompt, tags)
2. Run `ark skill list`
3. Output shows only `builtin` skills -- no `project` entries
4. Run `ark skill show project-bug-report`
5. Output: `Skill not found: project-bug-report`

### Analysis of root cause
`FileSkillStore` (`packages/core/stores/skill-store.ts:49-72`) is constructed with `builtinDir` and `userDir` but NO `projectDir` (`packages/core/app.ts:752-755`):
```
skills: asValue(new FileSkillStore({
  builtinDir: join(storeBaseDir, "skills"),
  userDir: join(this.config.arkDir, "skills"),
})),
```

The `list()` method (`skill-store.ts:60-71`) supports `projectRoot` as an argument and will scan `.ark/skills/` if passed, but the RPC handler (`packages/server/handlers/resource.ts:53`) calls `app.skills.list()` without passing `projectRoot`:
```
router.handle("skill/list", async () => ({ skills: app.skills.list() }));
```

So `projectDir` is never set during construction, and `projectRoot` is never passed at call time. Project-level skills are architecturally supported but never wired up.

### Workaround
Copy project skills to the global directory `~/.ark/skills/` instead. They will appear with source `global`. Example: `cp .ark/skills/project-bug-report.yaml ~/.ark/skills/`

## Bug-005: `ark session list` shows deleted sessions as "running" [FIXED]

**Verified:** 2026-04-13 on ark v0.13.0 / Claude Code v2.1.104 -- FIXED. Started session from `~` (non-git dir), expected original "crashes then stays running forever" symptom. Actual: (1) Ark falls back to running Claude directly in `~` instead of crashing (no worktree attempted), (2) after `ark session stop` + `ark session delete`, the session no longer appears in `ark session list`, (3) tmux session properly killed on stop. Soft-delete sets DB `status: deleting` (intentional 90s undo window), but list command correctly hides it. All original symptoms gone.

### Expected behavior
After `ark session stop <id>` and `ark session delete <id>`, the session should not appear in `ark session list`. If it does appear, it should show its actual status (`deleting`), not `running`.

### Steps to reproduce
1. `ark session start --repo . --summary "test" --flow bare --dispatch` from a non-git directory (`~`)
2. Claude Code exits immediately (no git repo = no worktree), tmux session dies
3. `ark session list` shows the session as `running` (status never updated from running)
4. `ark session stop <id>` → "Stopped"
5. `ark session delete <id>` → "Session deleted (undo available for 90s)"
6. `ark session list` still shows the session as `running`
7. `sqlite3 ~/.ark/ark.db` shows `status: deleting` -- but the list command renders it as `running`

### Analysis of root cause
Two issues:

1. **Status not updated on crash:** When Claude Code exits abnormally (tmux pane dies), there's no hook feedback to update session status. The conductor would handle this via hook status, but CLI mode runs with `skipConductor: true` (`packages/core/app.ts`). So status stays `running` forever.

2. **List shows `deleting` as `running`:** The `session list` CLI command (`packages/cli/commands/session.ts`) renders the status from the DB but may be reading a cached or pre-delete value. The soft-delete sets status to `deleting` but the list formatter doesn't handle this status, falling through to displaying the previous visual state.

### Workaround
Hard-delete from SQLite directly: `sqlite3 ~/.ark/ark.db "DELETE FROM sessions WHERE id='<id>';"` and kill orphaned tmux: `tmux kill-session -t ark-s-<id>`

## Bug-006: Task not auto-delivered to agent after dispatch -- three compounding issues [WORKS / PARTIAL]

**Verified:** 2026-04-13 on ark v0.13.0 / Claude Code v2.1.104 -- initial task IS delivered via the positional `claude` command-line argument in the launcher (NOT via the MCP channel, contrary to original analysis). Agent received "test-bug-001" as its first prompt and began executing. However, the MCP channel still reports "1 MCP server failed" and the dev-channels flag is ignored -- so steer/update messages sent AFTER the initial task may not deliver. Follow-up testing needed for post-initial-task message delivery.

### Expected behavior
After `ark session start --summary "Add CONTRIBUTING.md" --dispatch`, Claude Code should automatically receive the task summary as its first prompt and start working immediately.

### Steps to reproduce
1. `ark session start --repo . --summary "Add CONTRIBUTING.md" --flow bare --dispatch --attach`
2. Claude Code launches in tmux, shows idle prompt `>` -- no task delivered
3. Agent sits idle until user manually types the task

### Analysis of root cause
Three compounding issues in the task delivery chain:

**Issue A: Wrong channel.ts path in local provider (FIXED)**
`LocalProvider.buildChannelConfig()` (`packages/compute/providers/local/index.ts:139`) pointed to `core/channel.ts` but the file was moved to `core/conductor/channel.ts`. The MCP server couldn't start because the script didn't exist. Fixed by updating the path.

Also duplicated in `channelMcpConfig()` (`packages/core/claude/claude.ts:108`) -- but this function is never called for local dispatch because the local provider's `buildChannelConfig` takes precedence via `opts?.channelConfig ??` short-circuit (`claude.ts:129`).

**Issue B: Stale global ~/.mcp.json shadows worktree config**
`~/.mcp.json` and the project-root `.mcp.json` (tracked in git) both contained stale `ark-channel` entries from previous sessions/machines. Claude Code loads global MCP configs first, so the stale entry shadowed the correct worktree `.mcp.json`. Fixed by cleaning `~/.mcp.json`. The project `.mcp.json` should be added to `.gitignore`.

**Issue C: Claude Code v2.1.79 doesn't support `claude/channel` MCP notifications**
Even after fixing A and B (channel process runs, HTTP port listens, POST returns `ok`), the task still doesn't appear in Claude Code. The channel uses `notifications/claude/channel` -- an experimental MCP capability (`channel.ts:36`). Claude Code v2.1.79 does not support this protocol. The notification is accepted by the MCP SDK but Claude doesn't render it. This is the **final blocker** -- no fix possible without a Claude Code version that supports the channel protocol.

### Workaround
Use `ark session send <id> "your task"` after dispatch. This types directly into the tmux session via `tmux send-keys`, bypassing the MCP channel entirely. Or use `--attach` and type the task yourself.

## Bug-009: Completed sessions cannot be re-attached or resumed via CLI [OPEN]

**Verified:** 2026-04-13 on ark v0.13.0 / Claude Code v2.1.104. Reproduced with session `s-f26ab1` after the implementer agent called `report(completed)` and the flow advanced through `verify -> pr -> merge`.

### Expected behavior
After a session reaches `status: completed`, the user should be able to:
1. Re-attach to the still-living tmux session if it exists, OR
2. Re-launch Claude Code with `--resume <claude-session-id>` to continue the conversation
3. Issue a single command to do this (e.g. `ark session resume <id>` or `ark session reattach <id>`)

The Claude Code conversation transcript still exists on disk, the worktree is still intact, and the work may not actually be done -- "completed" here is a flow-state-machine concept, not a "done" signal from the user.

### Steps to reproduce
1. Dispatch any flow that has a terminal stage (e.g. `quick`, `fix-bug-quick`)
2. Wait for the agent to call `report(completed)` -- Ark advances the flow to its terminal node and sets `status: completed`
3. Some time passes (or laptop sleep) -- the tmux session dies on its own
4. `ark session attach s-f26ab1` -> "No active session. Dispatching..." then "Stage 'merge' is action, not agent" or "Not ready (status: completed)"
5. `ark session dispatch s-f26ab1` -> rejected (status not in `["ready", "blocked"]`)
6. `ark session resume s-f26ab1` -> guard at `services/session.ts` rejects: "Cannot resume a completed session"
7. `tmux attach -t ark-s-f26ab1` -> "can't find session"

### Analysis of root cause
Three compounding issues:

**Issue A: `attach` doesn't recover from db/tmux desync** (`packages/cli/commands/session.ts:316`)
```ts
if (!s.session_id) {
  console.log("No active session. Dispatching...");
  const r = await ark.sessionDispatch(id);  // tries dispatch, fails on terminal stage
}
```
`attach` checks the DB column `session_id` (tmux name), not whether `tmux has-session -t ark-<id>` is actually alive. If the DB lost the reference but tmux is still up, attach won't find it. If both are gone, attach falls back to dispatch which doesn't know how to handle terminal stages.

**Issue B: `dispatch` rejects completed sessions** (`packages/core/services/session-orchestration.ts:231`)
```ts
if (session.status !== "ready" && session.status !== "blocked") {
  return { ok: false, message: `Not ready (status: ${session.status}). Stop it first, or wait for it to finish.` };
}
```
There is no path that allows a completed session to be re-dispatched even with `--force`. The user has to manually patch the SQLite row.

**Issue C: `resume` explicitly forbids completed sessions** (`packages/core/services/session.ts:130`)
```ts
if (session.status === "completed") {
  return { ok: false, message: "Cannot resume a completed session" };
}
```
This is an intentional guard. The author treated "completed" as terminal and irreversible, but that's wrong -- the user may want to keep working on a feature that the flow's state machine considers "done".

**Issue D: `claude_session_id` not always populated**
Even after manually patching status back to `ready`, Bug-006 / dispatch lifecycle issues may leave `claude_session_id = NULL` in the DB. Without it, dispatch starts a fresh Claude Code session instead of `--resume`-ing the existing one. The user has to grep `~/.claude/projects/<encoded-workdir>/*.jsonl` to find the right UUID.

### Workaround
Manual SQLite patch + re-dispatch:
```bash
SESSION_ID=s-f26ab1
WORKTREE=$(sqlite3 ~/.ark/ark.db "SELECT workdir FROM sessions WHERE id='$SESSION_ID';")
SLUG=$(echo "$WORKTREE" | sed 's/\//-/g')
CLAUDE_ID=$(ls -t "$HOME/.claude/projects/$SLUG"/*.jsonl 2>/dev/null | head -1 | xargs basename | sed 's/\.jsonl$//')

sqlite3 ~/.ark/ark.db "UPDATE sessions SET status='ready', claude_session_id='$CLAUDE_ID' WHERE id='$SESSION_ID';"
ark session dispatch $SESSION_ID    # now uses claude --resume <id>
ark session attach $SESSION_ID
```

A proper fix would add `ark session reattach <id>` (or extend `ark session resume <id>`) that:
1. Removes the `completed` guard
2. Auto-discovers `claude_session_id` from `~/.claude/projects/<slug>/` if not in DB
3. Resets `status` to `ready`, clears `session_id`
4. Re-dispatches with `--resume`
5. Optionally attaches

## Bug-010: Release binary for darwin-arm64 has corrupt LC_CODE_SIGNATURE, killed by macOS Gatekeeper [OPEN / FIX IN PR #89]

**Verified:** 2026-04-14 on macOS Darwin 23.5.0 arm64 with ark v0.14.0 tarball downloaded fresh from the GitHub release.
**Fix status:** Client-side workaround landed in PR https://github.com/ytarasova/ark/pull/89 (commits 22b7cde, 19e5cbd). Release CI codesign step still pending as a follow-up.

### Expected behavior
After running the documented binary install, `ark --version` and `ark web` should run successfully.

### Steps to reproduce
1. On macOS 14+ arm64, run the documented curl installer for ark v0.14.0
2. Run `ark web`
3. Output: `zsh: killed     ark web`
4. Every ark invocation dies immediately with exit 137 (SIGKILL) before any output
5. `codesign -dvv ~/.ark/bin/ark` reports: "code object is not signed at all"
6. `codesign --force --sign - ~/.ark/bin/ark` fails with: "invalid or unsupported format for signature"

### Analysis of root cause
The ark binary in the v0.14.0 release tarball has a malformed LC_CODE_SIGNATURE load command. The Mach-O load command claims a 261,744-byte signature region, but the embedded SuperBlob's internal length field claims 601,394 bytes -- about 2.3x too large.

Verified with `otool -l` and raw byte inspection:
- LC_CODE_SIGNATURE load command: dataoff=76962640, datasize=261744
- Embedded at offset 76962640: magic=fade0cc0 (CSMAGIC_EMBEDDED_SIGNATURE, correct), length=0x00092d32=601394 (WRONG)

File size is exactly 76962640 + 261744 = 77224384, so the load command's view is internally consistent with the file. The embedded SuperBlob's length field is the corrupted value.

When codesign reads the binary, the magic is valid so it enters the signature parser, reads the bogus length, tries to parse 601KB of signature data, overruns the file, and reports "not signed at all". When codesign writes with --force --sign -, it tries to parse the existing blob first to know how much space is allocated, fails, reports "invalid or unsupported format".

When the kernel exec()s the binary, macOS 14+ on arm64 requires every executable to have a valid code signature. The kernel finds the LC_CODE_SIGNATURE load command, validation fails, process is killed with SIGKILL. User sees "zsh: killed", exit 137.

**Why only ark, not the bundled tmux?** The tmux binary in the same tarball has flags 0x20002 (adhoc, linker-signed) -- it was signed by its linker at build time. The ark binary is built with `bun build --compile`, which on macOS arm64 is a known Bun issue: it emits an LC_CODE_SIGNATURE load command with a placeholder SuperBlob that has an incorrect length field. The release CI does NOT run codesign after the Bun compile step.

This blocks ALL new macOS arm64 users from using the documented binary install path.

### Workaround
Strip the corrupt signature and re-sign ad-hoc (two separate codesign invocations):
1. `codesign --remove-signature ~/.ark/bin/ark` -- removes the broken LC_CODE_SIGNATURE entirely
2. `codesign --force --sign - ~/.ark/bin/ark` -- adds a clean ad-hoc signature
3. `~/.ark/bin/ark --version` -- now runs

`codesign --remove-signature` is the key command -- unlike `--force --sign -`, it does not attempt to parse the existing signature blob before removing it, so it tolerates the corrupt state.

A proper fix belongs in the release CI: after the Bun compile step, run `codesign --remove-signature` then `codesign --force --sign -` before tarballing. Better: upstream Bun should emit correct linker-signed binaries directly.

## Bug-011: install.sh follows symlinks, corrupts repo files on developer machines [OPEN / FIX IN PR #89]

**Verified:** 2026-04-14 on macOS Darwin 23.5.0 arm64. Reproduced when a developer has previously run `make install` (which creates `~/.ark/bin/ark` as a symlink to the source-tree ark wrapper) and then runs the curl installer.
**Fix status:** Fixed in PR https://github.com/ytarasova/ark/pull/89 (commits 22b7cde, 19e5cbd) by removing pre-existing symlinks under $BIN_DIR before the cp step.

### Expected behavior
Running the binary installer should replace only files under `~/.ark/` and must not modify any files outside that directory, especially not files in the user's source tree.

### Steps to reproduce
1. Clone the ark repo and run `make install` once -- creates `~/.ark/bin/ark` as a symlink pointing at the source-tree's ark bash wrapper
2. Run the curl installer
3. After install, the supposedly-untouched repo file is now a Mach-O binary instead of a bash script
4. `git status` in the repo shows ark as modified, containing the compiled binary blob

### Analysis of root cause
The install script's extraction step uses a plain recursive copy. On macOS, with the destination being an existing symlink, the copy tool follows the symlink and writes to the symlink's target. So writing the new ark binary to `~/.ark/bin/ark` (a symlink) actually writes the bytes into the developer's source-tree file.

This destroys the developer's source file, which may be tracked in git. Because it's a 77MB binary replacing a 4-line shell script, git status, diff tools, and the developer's editor all misbehave.

### Workaround
Either:
1. Before running the installer, delete the symlink first
2. Or use `git checkout -- ark` in the source repo to restore the bash wrapper after the bad install

A proper fix belongs in install.sh: remove all pre-existing symlinks inside `~/.ark/` before the copy step, and use copy flags that do not follow symlinks (the `-P` or `--no-dereference` variant depending on platform).

## Bug-012: install.sh calls `error` function before it is defined -- GitHub API rate limit produces "command not found" instead of useful message [OPEN]

**Verified:** 2026-04-15 on darwin/arm64 during live reproduction of Bug-010/Bug-011 via `curl -fsSL https://ytarasova.github.io/ark/install.sh | bash` without `ARK_VERSION` set.

### Expected behavior
If install.sh cannot determine the latest release version (e.g. the GitHub API returns 403 due to rate limiting on an unauthenticated curl), it should print a clear error message via the `error()` helper and exit with status 1.

### Steps to reproduce
1. Ensure `ARK_VERSION` environment variable is unset
2. Make sure the curl request to GitHub API is rate-limited (can happen after repeated runs in a short window, or from a shared IP)
3. Run: `curl -fsSL https://ytarasova.github.io/ark/install.sh | bash`
4. Observed output:
   ```
   curl: (22) The requested URL returned error: 403
   bash: line 32: error: command not found
   ```

### Analysis of root cause
In `docs/install.sh`, the helper functions `info()`, `warn()`, and `error()` are defined at lines 36-38. But the `error "Could not determine latest release..."` call at line 32 happens *before* that definition when the GitHub API call at lines 28-30 fails and VERSION ends up empty.

When bash executes a script read from a pipe (`curl | bash`) or stdin, it parses and runs commands sequentially. Function definitions are only registered once bash encounters them in the execution flow. So at line 32, bash has not yet seen the definition at line 38, and `error` resolves as an external command -- which does not exist -- yielding `command not found`.

The user sees two unhelpful error lines instead of the intended "Could not determine latest release. Use ARK_VERSION=v0.1.0 to pin." message.

### Workaround
Move the function definitions (`info`, `warn`, `error`) to the top of the script, immediately after `set -euo pipefail` and before the `USE_MAIN` parsing block. This is a 3-line relocation with no behavior change on the happy path.

Related: reproducing Bug-010/Bug-011 in a single install run today exposed this latent issue. It is independent of Bug-010 and Bug-011 and would deserve its own one-line fix in a follow-up PR.
