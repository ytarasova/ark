The OAuth implementation requires a provider abstraction layer in packages/core/auth/. Analysis of the existing codebase reveals a session-based authentication system in session.ts that can be extended with pluggable OAuth providers without disrupting the current flow.

The PKCE extension (RFC 7636) is mandatory for public clients. The implementation will use S256 challenge method exclusively -- plain challenges are deprecated and most providers reject them.

  2. Implement the GitHub provider in github.ts with full PKCE flow, including code verifier generation using crypto.randomBytes() and S256 challenge derivation
  3. Build a token store in token-store.ts with encrypted-at-rest storage, automatic refresh rotation, and replay detection
  4. Wire the new auth layer into the existing session bootstrap in session.ts, gated behind a feature flag
  5. Write comprehensive tests covering the full authorization flow, token refresh, and error cases

Starting with the provider abstraction. The OAuthProvider interface defines three methods that any provider must implement. This keeps the core auth system provider-agnostic while allowing easy addition of Google, GitLab, or Bitbucket providers later.

The interface is deliberately minimal. Authentication is a domain where simplicity directly correlates with security -- every additional method is a potential misconfiguration vector.

import { randomBytes, createHash } from "crypto";

export interface TokenSet {
  accessToken: string;
  refreshToken: string;
  expiresAt: number;
  scope: string;
}

export interface OAuthProvider {
  getAuthorizationUrl(state: string, challenge: string): string;
  exchangeCode(code: string, verifier: string): Promise<TokenSet>;
  getUserProfile(accessToken: string): Promise<UserProfile>;
}

export function generateVerifier(): string {
  return randomBytes(32).toString("base64url");
}

export function deriveChallenge(verifier: string): string {
  return createHash("sha256")
    .update(verifier)
    .digest("base64url");
}

The GitHub provider consumes the v2 endpoints as specified. Token refresh uses rotation -- each refresh produces a new refresh token, and the old one is immediately invalidated. This prevents token replay attacks if a refresh token is intercepted.

Token storage leverages the existing keychain integration. The TokenStore class wraps config.getSecret() and config.setSecret(), adding an in-memory LRU cache with a 5-minute TTL to avoid repeated keychain lookups.

packages/core/auth/token-store.ts new -- 67 lines 
export class TokenStore {
  private cache = new Map<string, CacheEntry>();

  async store(provider: string, tokens: TokenSet): Promise<void> {
    const key = `oauth_${provider}`;
    await config.setSecret(key, JSON.stringify(tokens));
    this.cache.set(key, {
      tokens,
      expiresAt: Date.now() + 5 * 60 * 1000,
    });
  }

  async retrieve(provider: string): Promise<TokenSet | null> {
    const key = `oauth_${provider}`;
    const cached = this.cache.get(key);
    if (cached && cached.expiresAt > Date.now()) {
      return cached.tokens;
    }
    const raw = await config.getSecret(key);
    return raw ? JSON.parse(raw) : null;
  }
}
