The current middleware is a single authenticateJWT function in src/middleware/auth.ts. It verifies the Bearer token and attaches the decoded user to req.user. There are 4 existing tests covering valid tokens, expired tokens, missing headers, and malformed tokens.

Here is the plan:

1. Create an auth strategy interface -- define AuthStrategy with a verify(req) method that returns a AuthResult. Both JWT and API key handlers will implement this.

2. Implement API key strategy -- check for X-API-Key header, look up the key in the database, and resolve to the associated user.

3. Refactor the middleware -- the main authenticate function tries strategies in order (JWT first if Bearer token present, then API key). Falls through to 401 if none succeed.

4. Add database schema for API keys -- table api_keys with columns: id, user_id, key_hash, name, created_at, last_used_at, expires_at.

5. Write tests -- 8 new test cases covering API key auth, strategy fallthrough, key expiry, and combined scenarios.

6. Update route registrations -- swap authenticateJWT for the new authenticate in all route files. Ensure backward compatibility by keeping authenticateJWT as an alias.

src/middleware/auth.ts export async function authenticate(req: Request, res: Response, next: NextFunction) { const strategies: AuthStrategy[] = [new JwtStrategy(), new ApiKeyStrategy()]; for (const strategy of strategies) { const result = await strategy.verify(req); if (result.authenticated) { req.user = result.user; return next(); } } return res.status(401).json({ error: "Authentication required" }); } // Backward compatibility export const authenticateJWT = authenticate;

The API key strategy checks the X-API-Key header, hashes the incoming key with SHA-256, and looks it up in the api_keys table. Keys have optional expiry dates and the last_used_at timestamp is updated on each successful auth.
