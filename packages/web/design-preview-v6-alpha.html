Planner

I will implement OAuth2 PKCE (Proof Key for Code Exchange) for the Ark web dashboard. This is the recommended auth flow for public clients -- it replaces the implicit grant flow which is now considered insecure.

The implementation covers four areas:

  • A new packages/core/auth/pkce.ts module that generates code verifiers and S256 challenges
  • An auth callback handler in packages/server/routes/auth-callback.ts that exchanges the authorization code
  • Token storage and refresh logic in packages/web/src/auth/token-store.ts
  • Session middleware update to validate tokens on protected routes

I will use the existing AppContext configuration for the OAuth provider URL and client ID. No new environment variables are needed -- the values come from the settings store.

The PKCE module uses crypto.subtle.digest for the S256 challenge and crypto.getRandomValues for the 128-byte code verifier. Both are available in Bun's global scope without imports.

The token store uses a simple Map<string, TokenEntry> with a 15-minute TTL and a background sweep interval. On expiry, it attempts a silent refresh using the stored refresh token before evicting.

The callback handler validates the state parameter against a stored nonce, exchanges the authorization code with the code verifier, and stores the resulting access and refresh tokens. The middleware checks the Authorization header, validates the token against the store, and injects the user context into the request.

I have wired the auth middleware into the route registration so all JSON-RPC endpoints require a valid token. The auth callback route itself is excluded from the middleware.

You

The token store should not use a background sweep. Use lazy eviction on access instead -- check TTL when the token is read and evict if expired. Background timers cause issues in tests and during shutdown.

Implementer

Good call. I will remove the setInterval sweep and switch to lazy eviction. Each get() call will check the token's expiry timestamp and attempt a refresh if within the grace window, or evict if fully expired.

Edited packages/core/auth/token-store.ts (+18 -31) 0.3s
Implementer

Updated. The token store is now stateless between accesses -- no background timers, no interval cleanup. The get() method checks expiresAt on every call. If the token has less than 60 seconds remaining and a refresh token exists, it refreshes inline. If the token is fully expired, it deletes the entry and returns null.

This is cleaner for testing since there are no dangling intervals to clear in afterAll hooks.
